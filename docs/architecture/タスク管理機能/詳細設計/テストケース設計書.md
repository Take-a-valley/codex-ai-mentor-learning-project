# テストケース設計書

## 1. 本書の目的

本書は、タスク管理機能の初期MVPに対するテストケースの方針と確認観点を整理することを目的とする。

要件定義、基本設計、API設計、DB設計、認証・認可設計、フロントエンド設計、バックエンド詳細設計をもとに、実装後に何を確認すべきかを明確にする。

## 2. 対象範囲

対象に含めるものは以下とする。

- バックエンド単体テスト
- フロントエンド単体テスト
- APIテスト
- 画面テスト
- 結合テスト
- 認証・認可テスト
- CSRFテスト
- 論理削除テスト
- 添付ファイルテスト
- 招待テスト
- パスワードリセットテスト
- 主要業務シナリオテスト

対象外とするものは以下とする。

- 負荷テスト
- セキュリティ診断
- 本番運用監視テスト
- CI/CDパイプラインテスト
- Wiki、WBS、チャット機能のテスト

## 3. テスト分類

初期MVPでは、以下の分類でテストを整理する。

| 分類 | 対象 |
|---|---|
| 単体テスト | Service、Repository、Controller、API、フロントエンドコンポーネント、画面単位の表示 |
| 結合テスト | 認証、認可、CSRF、DB、メール、添付ファイル、業務ルールを含む一連の処理 |
| シナリオテスト | 利用者の操作フロー単位の確認 |

APIテストと画面テストは、単体テストの一部として扱う。

認証、認可、CSRF、招待、パスワードリセット、論理削除など、複数要素が関係する確認は結合テストとして扱う。

## 4. テストの基本方針

テストでは、正常系だけでなく異常系を確認する。

特に以下を重点的に確認する。

- 認証が必要なAPIを未ログインで利用できないこと
- 権限のない操作ができないこと
- CSRFトークンなしで状態変更APIを実行できないこと
- 論理削除済みデータが通常取得で返らないこと
- PROJECT_ADMINが0人になるプロジェクトロール変更またはメンバー削除を禁止すること
- 添付ファイルの容量、種別制限が守られること
- 招待URL、パスワードリセットURLの期限が守られること

SYSTEM_ADMINが0人になる操作の確認は、システムロール変更機能を実装対象に含める時点でテストケース化する。

初期MVPでは、SYSTEM_ADMINのロール変更画面およびAPIは実装対象外とする。

SYSTEM_ADMINが0人になる操作の禁止は、システムロール変更機能を実装対象に含める時点でService層の業務ルールとして実装し、テストケース化する。

## 5. テストデータ方針

テストでは、以下のユーザーを用意する。

| ユーザー | システムロール | プロジェクトロール |
|---|---|---|
| システム管理者 | SYSTEM_ADMIN | PROJECT_ADMIN |
| 一般ユーザーA | USER | PROJECT_ADMIN |
| 一般ユーザーB | USER | MEMBER |
| 一般ユーザーC | USER | VIEWER |
| プロジェクト未参加ユーザー | USER | なし |
| 論理削除済みユーザー | USER | 任意 |

テストでは、以下のプロジェクトを用意する。

| プロジェクト | 用途 |
|---|---|
| 通常プロジェクト | 基本操作の確認 |
| 論理削除済みプロジェクト | 削除後の取得不可確認 |
| 複数管理者プロジェクト | 管理者ロール変更確認 |
| 管理者1人プロジェクト | 管理者0人禁止確認 |

Service層では以下を確認する。

| 対象 | 観点 |
|---|---|
| AuthService | ログイン成功、ログイン失敗、ログアウト |
| ProjectService | 作成、更新、削除、権限判定 |
| MemberService | メンバー一覧、ロール変更、管理者0人禁止 |
| TaskService | 作成、更新、Status変更、削除 |
| CommentService | 作成、投稿者本人の編集、管理者削除 |
| AttachmentService | 追加、取得、削除、容量制限、種別制限 |
| InvitationService | 招待作成、招待承認、期限切れ判定 |
| PasswordResetService | リセット申請、再設定、期限切れ判定 |

Repository層では以下を確認する。

- 論理削除済みデータを通常取得しないこと
- projectIdで対象データを絞り込めること
- tokenで招待情報、パスワードリセット情報を取得できること
- 有効なデータのみを取得できること

## 8. APIテスト観点

APIテストでは以下を確認する。

- 正しいリクエストで成功レスポンスが返ること
- 不正なリクエストで400が返ること
- 未ログインで認証必須APIを呼ぶと401が返ること
- 権限不足で403が返ること
- 対象データが存在しない場合に404が返ること
- 業務制約違反で409が返ること
- ファイルサイズ超過で413が返ること
- 許可されていないファイル種別で415が返ること
- 想定外エラーでは500が返ること
- エラーレスポンス形式が統一されていること

## 9. 認証テスト観点

認証では以下を確認する。

| ケース | 期待結果 |
|---|---|
| 正しいメールアドレスとパスワードでログイン | ログイン成功 |
| 誤ったパスワードでログイン | 401 |
| 存在しないメールアドレスでログイン | 401 |
| ログアウト | セッション無効化 |
| 未ログインで認証必須API呼び出し | 401 |
| セッション期限切れ後のAPI呼び出し | 401 |
| 論理削除済みユーザーのログイン | 401 |

## 10. CSRFテスト観点

CSRFでは以下を確認する。

| ケース | 期待結果 |
|---|---|
| CSRFトークン取得APIを呼び出す | tokenとheaderNameが返る |
| POSTでCSRFトークンあり | 成功 |
| POSTでCSRFトークンなし | 403 |
| PUTでCSRFトークンなし | 403 |
| PATCHでCSRFトークンなし | 403 |
| DELETEでCSRFトークンなし | 403 |
| GETでCSRFトークンなし | 成功 |
| 認証不要POSTでもCSRFトークンなし | 403 |

| タスク閲覧 | 可 | 可 | 可 | 可 | 不可 |
| タスク作成 | 可 | 可 | 可 | 不可 | 不可 |
| タスク編集 | 可 | 可 | 可 | 不可 | 不可 |
| タスク削除 | 可 | 可 | 不可 | 不可 | 不可 |
| コメント作成 | 可 | 可 | 可 | 不可 | 不可 |
| コメント編集 | 投稿者本人のみ | 投稿者本人のみ | 投稿者本人のみ | 不可 | 不可 |

## 12. プロジェクト管理テスト観点

プロジェクト管理では以下を確認する。

- SYSTEM_ADMINのみプロジェクト作成できること
- プロジェクト作成直後、作成者がPROJECT_ADMINになること
- プロジェクト情報を更新できること
- プロジェクトを論理削除できること
- 論理削除済みプロジェクトが一覧に表示されないこと
- プロジェクト削除時にタスク、コメント、添付ファイルも論理削除扱いになること

## 13. メンバー管理テスト観点

メンバー管理では以下を確認する。

- プロジェクトメンバー一覧を取得できること
- PROJECT_ADMINがメンバーのロールを変更できること
- SYSTEM_ADMINが任意のユーザーをPROJECT_ADMINに委任できること
- MEMBERはロール変更できないこと
- VIEWERはロール変更できないこと
- PROJECT_ADMINが0人になるロール変更を禁止すること
- PROJECT_ADMINが0人になるメンバー削除を禁止すること
- メンバー削除は論理削除であること
- 論理削除されたメンバーが通常のメンバー一覧に表示されないこと

## 14. 招待テスト観点

招待では以下を確認する。

- PROJECT_ADMINまたはSYSTEM_ADMINが招待を作成できること
- 招待メールがMailpitで確認できること
- 招待URLから招待情報を確認できること
- 未登録ユーザーが招待URLから登録後に参加できること
- 登録済みユーザーが招待URLから参加できること
- 招待時の初期ロールがMEMBERになること
- 期限切れ招待URLは使用できないこと
- 使用済み招待URLは再利用できないこと
- 招待情報が履歴として残ること
- 招待先メールアドレスとログインユーザーのメールアドレスが一致しない場合は承認できないこと

## 15. パスワードリセットテスト観点

パスワードリセットでは以下を確認する。

- パスワードリセット申請ができること
- 登録済みメールアドレスにリセットメールが送信されること
- 未登録メールアドレスでも同じレスポンスを返すこと
- リセットURLからパスワード再設定できること
- 期限切れリセットURLは使用できないこと
- 使用済みリセットURLは再利用できないこと
- パスワード再設定後、新しいパスワードでログインできること
- パスワードリセット情報が履歴として残ること

## 16. タスク管理テスト観点

タスク管理では以下を確認する。

- タスク一覧をStatus別に取得できること
- タスクを作成できること
- タスクを編集できること
- タスクを削除できること
- タスク削除はPROJECT_ADMINまたはSYSTEM_ADMINのみ可能であること
- Statusを `TODO`、`IN_PROGRESS`、`REVIEW`、`DONE` に変更できること
- Priorityを `HIGHEST`、`HIGH`、`MEDIUM`、`LOW`、`LOWEST` として扱えること
- ドラッグアンドドロップ後にStatus変更APIが呼ばれること
- 同一Status内の並び替えは初期MVP対象外であること

## 17. コメントテスト観点

コメントでは以下を確認する。

- コメントを作成できること
- コメント一覧を取得できること
- コメント投稿者本人が自分のコメントを編集できること
- 他人のコメントを編集できないこと
- PROJECT_ADMINまたはSYSTEM_ADMINがコメントを削除できること
- MEMBERはコメントを削除できないこと
- VIEWERはコメントを閲覧できること
- 削除済みユーザーの表示名が「削除済ユーザー」になること

## 18. 添付ファイルテスト観点

添付ファイルでは以下を確認する。

- 添付ファイルを追加できること
- 添付ファイル一覧を取得できること
- 添付ファイルをダウンロードできること
- 添付ファイルを削除できること
- 1ファイル5MBを超える場合は登録できないこと
- 許可されていないファイル種別は登録できないこと
- 添付ファイル本体がDBに保存されること
- 論理削除済み添付ファイルは通常取得されないこと

## 19. 論理削除テスト観点

論理削除では以下を確認する。

- ユーザー削除時、作成済みタスクやコメントは削除されないこと
- 削除済みユーザーの表示名が「削除済ユーザー」になること
- 論理削除済みデータが通常の詳細取得で取得できないこと
- プロジェクト削除時、タスク、コメント、添付ファイルが論理削除扱いになること
- 論理削除済みタスクが通常一覧に表示されないこと
- 論理削除済みコメントが通常一覧に表示されないこと
- 論理削除済み添付ファイルが通常一覧に表示されないこと
- 招待情報とパスワードリセット情報は履歴として残ること

## 20. 画面テスト観点

画面では以下を確認する。

- 未認証画面で認証後レイアウトが表示されないこと
- 認証後画面でヘッダー、サイドバーが表示されること
- 権限に応じてボタンが表示、非表示になること
- 入力エラーがフォーム付近に表示されること
- APIエラーが適切な位置に表示されること
- ローディング表示が出ること
- 送信中ボタンが無効化されること
- 空状態表示が出ること
- タスク作成・編集モーダルが開閉できること
- カンバン画面でStatus列が表示されること

## 21. 主要シナリオテスト

主要シナリオとして以下を確認する。

### 21.1 プロジェクト作成からタスク作成まで

1. SYSTEM_ADMINでログインする
2. プロジェクトを作成する
3. 作成したプロジェクトのPROJECT_ADMINになっていることを確認する
4. カンバン画面を開く
5. タスクを作成する
6. ToDo列にタスクが表示されることを確認する

### 21.2 メンバー招待からタスク操作まで

1. PROJECT_ADMINでログインする
2. メンバーをメール招待する
3. Mailpitで招待メールを確認する
4. 招待URLからユーザー登録または参加を行う
5. 招待ユーザーがMEMBERとして参加していることを確認する
6. MEMBERがタスクを作成できることを確認する
7. MEMBERがタスク削除できないことを確認する

### 21.3 タスク詳細、コメント、添付ファイル

1. MEMBERでログインする
2. タスク詳細画面を開く
3. コメントを投稿する
4. 添付ファイルを追加する
5. コメントと添付ファイルが表示されることを確認する
6. PROJECT_ADMINでログインする
7. コメント削除、添付ファイル削除ができることを確認する

### 21.4 パスワードリセット

1. パスワードリセット申請画面を開く
2. 登録済みメールアドレスで申請する
3. Mailpitでリセットメールを確認する
4. リセットURLから新しいパスワードを設定する
5. 新しいパスワードでログインできることを確認する

## 22. テスト優先度

初期MVPでは以下を優先して確認する。

| 優先度 | 対象 |
|---|---|
| 高 | 認証、認可、CSRF、プロジェクト作成、タスク作成・編集・削除 |
| 高 | 管理者が0人になる操作の禁止 |
| 高 | 論理削除 |
| 中 | コメント、添付ファイル |
| 中 | 招待、パスワードリセット |
| 中 | フロントエンド表示制御 |
| 低 | 詳細なレスポンシブ表示 |
| 低 | 細かいUI文言 |

## 23. 決定事項

- 正常系と異常系の両方を確認する
- 認証、認可、CSRFを重点的に確認する
- 管理者が0人になる操作を重点的に確認する
- 削除は論理削除として確認する
- 添付ファイルはDB保存、5MB制限、ファイル種別制限を確認する
- 招待URLの有効期限は24時間として確認する
- パスワードリセットURLの有効期限は30分として確認する
- 画面表示制御は補助であり、最終認可はバックエンドで確認する
