# 実装計画書

## 1. 目的

本書は、タスク管理機能の初期MVPを実装するための作業順序、実装範囲、確認観点を整理することを目的とする。

要件定義、基本設計、詳細設計で決定した内容をもとに、実装フェーズで迷わず作業を進められるようにする。

## 2. 実装対象

初期MVPでは、複数プロジェクト対応のカンバン方式タスク管理機能を実装する。

対象機能は以下とする。

- メールアドレスとパスワードによるログイン
- セッション方式による認証
- CSRF対策
- パスワードリセット
- システム管理者、プロジェクト管理者、メンバー、閲覧者の権限管理
- プロジェクト作成、編集、削除
- プロジェクトメンバー招待、参加、ロール変更、削除
- タスク作成、編集、Status変更、削除
- カンバン画面でのドラッグアンドドロップによるStatus変更
- コメント作成、編集、削除
- 添付ファイル追加、取得、削除
- 主要データの論理削除
- 招待メール、パスワードリセットメールの開発環境確認

## 3. 実装対象外

初期MVPでは、以下は実装対象外とする。

- Wiki機能
- WBS、ガントチャート機能
- プロジェクト内チャット機能
- 本番デプロイ
- CI/CD
- 本番向けメール送信方式
- 外部ストレージ連携
- ログイン失敗回数制限
- ステータス列のカスタマイズ
- アカウント復旧機能
- 論理削除済みユーザーと同じメールアドレスでの再登録

## 4. 実装方針

実装は、一般的な開発順序に準じて、環境、DB、バックエンド、フロントエンド、テストの順に進める。

最初に動作基盤を整え、その後に業務機能を積み上げる。

実装順序は以下を基本とする。

1. 開発環境構築
2. DB・マイグレーション実装
3. Spring Boot基盤実装
4. Entity実装
5. Repository実装
6. 共通例外実装
7. Security実装
8. Service実装
9. Controller実装
10. フロントエンド基盤実装
11. APIクライアント実装
12. 画面実装
13. テスト実装
14. 動作確認

## 5. 開発環境構築

開発環境ではDocker Composeを使用する。

Docker Composeでは以下を起動する。

- MySQL 8系
- Mailpit

フロントエンドとバックエンドは、初期段階ではローカルPC上で起動する。

この方針は、初期実装時の学習容易性とトラブル切り分けのしやすさを優先するためである。
本リポジトリは将来的にポートフォリオとして公開することを想定するため、アプリケーション実装が安定した後に、フロントエンドとバックエンドのDocker化を追加検討する。

確認事項は以下とする。

- MySQLコンテナが起動できること
- Mailpitコンテナが起動できること
- MySQLへ接続できること
- MailpitのWeb UIを開けること
- `.env` がGit管理対象外であること
- `.env.example` が用意されていること

## 6. DB・マイグレーション実装

DB変更管理にはFlywayを使用する。

まず、設計済みのテーブルをマイグレーションファイルとして作成する。

実装順序は以下とする。

1. ユーザー系テーブル
2. プロジェクト系テーブル
3. タスク系テーブル
4. コメント、添付ファイル系テーブル
5. 招待、パスワードリセット系テーブル
6. インデックス、制約
7. 開発用初期データ

主要データは論理削除を前提とする。

招待情報とパスワードリセット情報は履歴として残す。

期限切れ状態はバッチ更新ではなく、参照時に `expires_at` と現在日時を比較して判定する。

## 7. バックエンド基盤実装

バックエンドはSpring Bootで実装する。

最初に以下の基盤を作成する。

- パッケージ構成
- 共通レスポンス形式
- 共通例外形式
- 共通例外ハンドラ
- CORS設定
- Security設定
- CSRF設定
- セッション設定
- JPA設定
- Flyway設定
- メール送信設定

認証方式はセッション方式とする。

セッション有効期限は最終アクセスから120分とする。

CSRF対策はSpring Securityで行い、CSRFトークン取得APIを用意する。

## 8. Entity実装

DB設計、Entity設計に従ってEntityを実装する。

対象Entityは以下とする。

- User
- Project
- ProjectMember
- Task
- TaskComment
- TaskAttachment
- ProjectInvitation
- PasswordResetToken

Entityでは以下を意識する。

- DBテーブルとの対応
- 主キー
- 外部キー
- Enum項目
- 作成日時
- 更新日時
- 削除日時
- 論理削除
- リレーション

通常の取得では、論理削除済みデータを除外する。

## 9. Repository実装

Repositoryでは、DBアクセスに専念する。

業務ルールはRepositoryではなくServiceで扱う。

主な実装対象は以下とする。

- ID検索
- 一覧検索
- 論理削除されていないデータの検索
- プロジェクト単位の検索
- email検索
- token検索
- status検索
- 期限切れ候補検索

Repositoryでは、必要に応じてSpring Data JPAのクエリメソッドを使用する。

複雑な検索が必要になった場合は、JPQLまたはカスタムRepositoryを検討する。

## 10. Service実装

Serviceでは、業務ルール、トランザクション、認可判定を扱う。

主なServiceは以下とする。

- AuthService
- PasswordResetService
- ProjectService
- ProjectMemberService
- ProjectInvitationService
- TaskService
- TaskCommentService
- TaskAttachmentService
- UserService

Serviceでは以下を必ず確認する。

- ログインユーザーが操作可能か
- 対象データが存在するか
- 対象データが論理削除されていないか
- 対象プロジェクトに参加しているか
- 必要なロールを持っているか
- 管理者が0人にならないか
- 添付ファイルの容量制限を満たすか
- 添付ファイルの種別制限を満たすか
- tokenが有効か
- tokenが期限切れでないか
- tokenが使用済みでないか

プロジェクト作成はシステム管理者のみ可能とする。

プロジェクト作成直後は、作成したシステム管理者を対象プロジェクトのプロジェクト管理者として登録する。

ProjectInvitationServiceでは、未登録ユーザーが招待URLから参加する場合のユーザー登録、プロジェクト参加、初期ロール付与まで扱う。

## 11. Controller実装

ControllerではHTTPリクエストを受け取り、Serviceを呼び出す。

Controllerには業務ルールを持たせない。

主なControllerは以下とする。

- AuthController
- CsrfController
- PasswordResetController
- ProjectController
- ProjectMemberController
- ProjectInvitationController
- TaskController
- TaskCommentController
- TaskAttachmentController

Controllerでは以下を整理する。

- APIパス
- HTTPメソッド
- 認証要否
- CSRF要否
- Request DTO
- Response DTO
- 呼び出すService
- HTTPステータス

状態変更APIではCSRFトークンを必須とする。

## 12. フロントエンド基盤実装

フロントエンドはNext.js、TypeScript、App Routerで実装する。

UIはTailwind CSSを基本とし、必要に応じてshadcn/uiを使用する。

サーバー状態はTanStack Queryで管理する。

画面状態はReact標準の状態管理を使用する。

最初に以下を実装する。

- ディレクトリ構成
- レイアウト
- 共通APIクライアント
- Query Hooks
- 認証状態取得
- CSRFトークン取得
- 共通エラー処理
- 共通UI部品
- ルーティング

## 13. フロントエンド画面実装

画面は以下の順に実装する。

1. ログイン画面
2. パスワードリセット申請画面
3. パスワード再設定画面
4. プロジェクト一覧画面
5. プロジェクト作成画面
6. プロジェクト編集画面
7. メンバー管理画面
8. 招待参加画面
9. カンバン画面
10. タスク作成モーダル
11. タスク編集モーダル
12. タスク詳細画面
13. コメント表示、作成、編集
14. 添付ファイル表示、追加、削除

タスク作成、タスク編集はモーダルで実装する。

カンバン画面では、ドラッグアンドドロップによるStatus変更を実装する。

ボタンやメニューは権限に応じて表示制御する。

ただし、最終的な認可判定はバックエンドで行う。

## 14. テスト実装

テストは、単体テストと結合テストに分けて実施する。

単体テストでは以下を確認する。

- Serviceの業務ルール
- Repositoryの検索条件
- DTOの入力チェック
- フロントエンドコンポーネント
- Query Hooks
- APIクライアント

結合テストでは以下を確認する。

- 認証
- 認可
- CSRF
- パスワードリセット
- 招待
- プロジェクト管理
- メンバー管理
- タスク管理
- コメント
- 添付ファイル
- 論理削除

E2Eテストでは、主要なユーザー操作を確認する。

## 15. 動作確認シナリオ

最低限、以下のシナリオを確認する。

### 15.1 ログイン

1. ログイン画面を開く
2. メールアドレスとパスワードを入力する
3. ログインできることを確認する
4. ログアウトできることを確認する

### 15.2 プロジェクト作成

1. システム管理者でログインする
2. プロジェクトを作成する
3. 作成者がPROJECT_ADMINとして登録されることを確認する
4. プロジェクト一覧に表示されることを確認する

### 15.3 メンバー招待

1. システム管理者またはプロジェクト管理者でログインする
2. メールアドレスを指定して招待する
3. Mailpitで招待メールを確認する
4. 招待URLから参加する
5. 初期ロールがMEMBERであることを確認する

### 15.4 タスク管理

1. カンバン画面を開く
2. タスクを作成する
3. タスクを編集する
4. ドラッグアンドドロップでStatusを変更する
5. タスクを削除する
6. 削除済みタスクが通常一覧に表示されないことを確認する

### 15.5 コメント

1. タスク詳細を開く
2. コメントを作成する
3. 自分のコメントを編集する
4. プロジェクト管理者がコメントを削除する
5. 削除済みコメントが通常表示されないことを確認する

### 15.6 添付ファイル

1. タスク詳細を開く
2. 添付ファイルを追加する
3. 添付ファイルを取得する
4. 添付ファイルを削除する
5. 5MBを超えるファイルが追加できないことを確認する
6. 許可されていないファイル種別が追加できないことを確認する

### 15.7 パスワードリセット

1. パスワードリセット申請画面を開く
2. メールアドレスを入力する
3. Mailpitでリセットメールを確認する
4. リセットURLから新しいパスワードを設定する
5. 新しいパスワードでログインできることを確認する

## 16. 実装時の注意点

実装時は以下に注意する。

- 設計書にない仕様を勝手に追加しない
- 認可判定をフロントエンドだけに任せない
- Service層で業務ルールを確認する
- Repositoryに業務判断を持たせない
- Controllerに業務判断を持たせない
- パスワード、token、セッションIDをログに出力しない
- 添付ファイル本体はDBに保存する
- 削除は物理削除ではなく論理削除を基本とする
- 招待情報とパスワードリセット情報は履歴として残す
- 期限切れは参照時に判定する
- システム管理者またはプロジェクト管理者が0人になる操作を禁止する
- 実装中に設計変更が必要になった場合は、先に設計書へ反映する

実装開始前に以下を確認する。

- 要件定義書が作成済みである

- 詳細設計書一式が作成済みである
- DB設計が確定している
- API設計が確定している
- 画面設計が確定している
- 認証・認可方針が確定している
- Docker Compose方針が確定している
- Flyway方針が確定している
- 実装対象外が明確になっている
- 開発環境で使用する技術バージョンが整理されている

## 18. 最初に着手する作業

最初に着手する作業は、開発環境構築とする。

具体的には以下から開始する。

1. リポジトリ内の実装用ディレクトリ構成を決める
2. Docker Compose設定を作成する
3. MySQLコンテナを起動する
4. Mailpitコンテナを起動する
5. Spring Bootプロジェクトを作成する
6. Next.jsプロジェクトを作成する
7. Spring BootからMySQLへ接続する
8. Flywayを有効化する
9. 開発用初期システム管理者データの投入方法を用意する

ポートフォリオ公開に向けたDocker化は、初期MVPの主要機能実装後に以下の順で検討する。

1. backend用Dockerfileを作成する
2. frontend用Dockerfileを作成する
3. Docker Composeにbackendサービスを追加する
4. Docker Composeにfrontendサービスを追加する
5. Docker Composeでfrontend、backend、mysql、mailpitをまとめて起動確認する
6. READMEにDockerによる起動手順を記載する

## 20. 完了条件

実装フェーズの初期MVP完了条件は以下とする。

- ログイン、ログアウトができる
- パスワードリセットができる
- システム管理者がプロジェクトを作成できる
- プロジェクト作成者がPROJECT_ADMINになる
- プロジェクト管理者を委任できる
- メンバーをメール招待できる
- 招待URLからプロジェクトに参加できる
- カンバン画面でタスクを管理できる
- ドラッグアンドドロップでStatusを変更できる
- コメントを管理できる
- 添付ファイルを管理できる
- 権限のない操作が拒否される
- 論理削除が正しく動作する
- 主要な正常系、異常系テストが通る
