# フロントエンド設計書

## 1. 本書の目的

本書は、タスク管理機能におけるフロントエンドの基本設計を整理することを目的とする。

要件定義書、基本設計書、画面設計書、API設計書、認証・認可設計書をもとに、Next.jsで構築する画面構成、ルーティング、コンポーネント方針、状態管理方針、API連携方針、ロール別表示制御を定義する。

## 2. 対象範囲

本書では、初期MVPであるカンバン方式タスク管理機能のフロントエンドを対象とする。

対象画面は以下とする。

- ログイン画面
- パスワードリセット申請画面
- パスワード再設定画面
- プロジェクト一覧画面
- プロジェクト作成画面
- プロジェクト編集画面
- カンバン画面
- タスク詳細画面
- メンバー管理画面
- メンバー招待画面
- 招待参加画面

以下は初期MVPでは対象外とする。

- Wiki画面
- WBS、ガントチャート画面
- プロジェクト内チャット画面
- 通知画面
- ステータス列カスタマイズ画面
- 外部ストレージ連携画面

## 3. 使用技術

| 分類 | 技術 |
|---|---|
| フロントエンド | Next.js 16.x系または作成時点のlatest |
| 言語 | TypeScript |
| 実行環境 | Node.js 24.14.1 |
| パッケージ管理 | npm 11.11.0 |
| API連携 | REST API |
| 認証方式 | セッション方式 |
| データ形式 | JSON |
| ファイルアップロード | multipart/form-data |
| UI | Tailwind CSS、必要に応じてshadcn/ui |
| 状態管理 | サーバー状態はTanStack Query、画面状態はReact標準 |

## 4. フロントエンドの責務

フロントエンドは以下を担当する。

- 画面表示
- ユーザー入力の受付
- 入力値の基本チェック
- API呼び出し
- APIレスポンスの画面反映
- 画面遷移
- ローディング表示
- エラー表示
- ロールに応じたボタンやリンクの表示制御
- モーダル、確認ダイアログの表示制御

ただし、認証・認可・業務ルールの最終判定はバックエンドで行う。

フロントエンド側の表示制御は、ユーザー体験向上のための補助とする。

## 5. フロントエンド構成方針

フロントエンドは、画面単位、機能単位、共通UI単位で構成する。

画面単位では、URLに対応する画面を管理する。

機能単位では、カンバン、タスク、コメント、添付ファイル、メンバー管理などのまとまりでコンポーネントを整理する。

共通UI単位では、ボタン、入力欄、モーダル、確認ダイアログなど、複数画面で再利用する部品を整理する。

## 6. ルーティング方針

初期MVPでは、以下のURL構成を想定する。

| 画面 | パス |
|---|---|
| ログイン画面 | /login |
| パスワードリセット申請画面 | /password-reset/request |
| パスワード再設定画面 | /password-reset/confirm?token=xxx |
| プロジェクト一覧画面 | /projects |
| プロジェクト作成画面 | /projects/create |
| プロジェクト編集画面 | /projects/{projectId}/edit |
| カンバン画面 | /projects/{projectId}/board |
| タスク詳細画面 | /projects/{projectId}/tasks/{taskId} |
| メンバー管理画面 | /projects/{projectId}/members |
| メンバー招待画面 | /projects/{projectId}/invitations/new |
| 招待参加画面 | /invitations/{token} |

認証が必要な画面では、ログイン状態を確認する。

未ログイン状態で認証必須画面へアクセスした場合、ログイン画面へ誘導する。

## 7. レイアウト方針

画面は、未認証画面と認証後画面でレイアウトを分ける。

### 未認証画面

未認証画面では、ログイン、パスワードリセット、招待参加など、認証前の操作を行う。

対象画面は以下とする。

- ログイン画面
- パスワードリセット申請画面
- パスワード再設定画面
- 招待参加画面

### 認証後画面

認証後画面では、共通レイアウトを使用する。

| 領域 | 内容 |
|---|---|
| ヘッダー | アプリ名、ログインユーザー情報、ログアウト |
| サイドバー | プロジェクト一覧、現在のプロジェクト |
| メイン領域 | 各画面の主要コンテンツ |
| モーダル | タスク作成・編集、確認操作 |

## 8. 画面構成方針

### ログイン画面

メールアドレスとパスワードを入力し、ログインAPIを呼び出す。

ログイン成功時はプロジェクト一覧画面へ遷移する。

ログイン失敗時は、メールアドレスまたはパスワードが正しくない旨を表示する。

### パスワードリセット申請画面

メールアドレスを入力し、パスワードリセット申請APIを呼び出す。

対象メールアドレスの存在有無に関わらず、同じ完了メッセージを表示する。

### パスワード再設定画面

URLのtokenを使用し、新しいパスワードを設定する。

パスワード再設定完了後はログイン画面へ遷移する。

### プロジェクト一覧画面

ログインユーザーが閲覧可能なプロジェクト一覧を表示する。

システム管理者は全プロジェクトを表示する。

通常ユーザーは参加済みプロジェクトのみ表示する。

### プロジェクト作成画面

システム管理者が新規プロジェクトを作成する。

プロジェクト作成直後は、作成したシステム管理者が対象プロジェクトのプロジェクト管理者となる。

作成完了後、作成したプロジェクトのカンバン画面へ遷移する。

### プロジェクト編集画面

システム管理者または対象プロジェクトのプロジェクト管理者が、プロジェクト情報を編集する。

### カンバン画面

タスクをStatus別に表示する。

Statusは以下とする。

- ToDo
- In Progress
- Review
- Done

タスク作成・編集はモーダルで行う。

ドラッグアンドドロップによるStatus変更を初期MVPに含める。

### タスク詳細画面

タスク詳細、コメント、添付ファイルを表示する。

タスク編集はモーダルから行う。

コメント作成、コメント編集、添付ファイル追加、添付ファイル削除はロールに応じて表示制御する。

### メンバー管理画面

プロジェクトメンバー一覧を表示する。

システム管理者または対象プロジェクトのプロジェクト管理者は、ロール変更とメンバー削除を行える。

プロジェクト管理者の委任は、プロジェクト作成後にメンバー管理画面で行う。

プロジェクト管理者が0人になる操作は実行できない。

### メンバー招待画面

メールアドレスを指定してプロジェクトへユーザーを招待する。

招待時の初期ロールはMEMBER固定とする。

### 招待参加画面

招待URLのtokenから招待情報を取得する。

未登録ユーザーは登録フォームを表示する。

登録済みユーザーはログイン後に招待承認を行う。

## 9. コンポーネント方針

フロントエンドでは、以下のような単位でコンポーネントを分ける。

| 種別 | 例 |
|---|---|
| レイアウト | AuthLayout, AppLayout |
| 共通UI | Button, Input, Select, Modal, ConfirmDialog |
| 認証 | LoginForm, PasswordResetForm |
| プロジェクト | ProjectList, ProjectForm |
| カンバン | KanbanBoard, KanbanColumn, TaskCard |
| タスク | TaskModal, TaskDetail |
| コメント | CommentList, CommentForm |
| 添付ファイル | AttachmentList, AttachmentUpload |
| メンバー | MemberTable, RoleSelect |
| 招待 | InvitationForm, InvitationJoin |

共通UIコンポーネントには、業務固有の処理を持たせない。

業務固有の処理は、画面または機能コンポーネント側で扱う。

## 10. 状態管理方針

フロントエンドでは、以下の状態を扱う。

| 状態 | 内容 |
|---|---|
| 認証状態 | ログインユーザー情報 |
| プロジェクト状態 | プロジェクト一覧、選択中プロジェクト |
| タスク状態 | カンバン表示用タスク一覧、タスク詳細 |
| メンバー状態 | メンバー一覧、ロール変更対象 |
| UI状態 | モーダル表示、確認ダイアログ、ローディング、エラー |
| CSRF状態 | CSRFトークン |

サーバー状態はTanStack Queryで管理し、画面状態はReact標準の状態管理で扱う。

## 11. API連携方針

フロントエンドは、バックエンドのREST APIを呼び出す。

APIレスポンスはJSON形式を基本とする。

添付ファイル取得APIのみ、ファイルデータとして扱う。

セッション方式を採用するため、API通信時にはCookieを送信する。

状態変更APIではCSRFトークンを送信する。

## 12. APIエラー制御方針

APIエラー時は、HTTPステータスに応じて画面表示を切り替える。

| HTTPステータス | 画面制御 |
|---|---|
| 400 | 入力内容を確認するメッセージを表示する |
| 401 | ログイン画面へ誘導する |
| 403 | 権限がない旨を表示する |
| 404 | 対象データが存在しない旨を表示する |
| 409 | 業務制約により操作できない旨を表示する |
| 413 | ファイルサイズ上限を超えている旨を表示する |
| 415 | 許可されていないファイル種別である旨を表示する |
| 500 | サーバーエラーが発生した旨を表示する |

## 13. 認証状態の扱い

認証後の画面表示時には、ログインユーザー情報取得APIを呼び出す。

ログインユーザー情報が取得できた場合、認証済みとして扱う。

401が返却された場合、未ログインとしてログイン画面へ誘導する。

ログアウト時はログアウトAPIを呼び出し、成功後にログイン画面へ遷移する。

## 14. CSRFトークンの扱い

フロントエンドは、ログイン画面表示時またはアプリ初期表示時にCSRFトークンを取得する。

取得したCSRFトークンは、POST、PUT、PATCH、DELETEのリクエスト時に送信する。

CSRFトークンの取得API、保存方法、ヘッダー名は後続工程で具体化する。

## 15. ロール別表示制御

フロントエンドでは、ログインユーザーのシステムロールと対象プロジェクト内ロールに応じて、操作ボタンやリンクの表示を制御する。

| 操作 | 表示条件 |
|---|---|
| プロジェクト作成 | システム管理者 |
| プロジェクト編集 | システム管理者または対象プロジェクトのプロジェクト管理者 |
| メンバー招待 | システム管理者または対象プロジェクトのプロジェクト管理者 |
| ロール変更 | システム管理者または対象プロジェクトのプロジェクト管理者 |
| タスク作成 | システム管理者、対象プロジェクトのプロジェクト管理者、メンバー |
| タスク編集 | システム管理者、対象プロジェクトのプロジェクト管理者、メンバー |
| タスク削除 | システム管理者または対象プロジェクトのプロジェクト管理者 |
| コメント作成 | システム管理者、対象プロジェクトのプロジェクト管理者、メンバー |
| コメント削除 | システム管理者または対象プロジェクトのプロジェクト管理者 |
| 添付ファイル追加 | システム管理者、対象プロジェクトのプロジェクト管理者、メンバー |
| 添付ファイル削除 | システム管理者または対象プロジェクトのプロジェクト管理者 |

画面側の表示制御は補助であり、バックエンドAPI側でも必ず認可チェックを行う。

## 16. 入力チェック方針

フロントエンドでは、ユーザー操作時に基本的な入力チェックを行う。

| 項目 | チェック内容 |
|---|---|
| メールアドレス | 必須、メール形式 |
| パスワード | 必須 |
| プロジェクト名 | 必須 |
| Summary | 必須 |
| Due date | 未来日 |
| Priority | 必須 |
| コメント本文 | 必須 |
| 添付ファイル | サイズ上限、ファイル種別 |

ただし、最終的な入力チェックはバックエンドでも行う。

## 17. ローディング表示方針

API通信中はローディング状態を表示する。

一覧取得中は一覧領域にローディング表示を行う。

作成、更新、削除などの状態変更中は、二重送信を防ぐために実行ボタンを無効化する。

ドラッグアンドドロップによるStatus変更中は、対象タスクカードを更新中として扱う。

## 18. 空状態表示方針

データが存在しない場合は、空状態を表示する。

| 対象 | 表示内容 |
|---|---|
| プロジェクト一覧 | 表示できるプロジェクトがない旨 |
| カンバン列 | 対象Statusにタスクがない旨 |
| コメント一覧 | コメントがない旨 |
| 添付ファイル一覧 | 添付ファイルがない旨 |
| メンバー一覧 | メンバーがいない旨 |

## 19. 確認ダイアログ方針

以下の操作では、実行前に確認ダイアログを表示する。

- プロジェクト削除
- タスク削除
- コメント削除
- 添付ファイル削除
- メンバー削除
- ロール変更

確認後にAPIを呼び出す。

キャンセル時はAPIを呼び出さない。

## 20. ファイルアップロード方針

添付ファイル追加時は、ファイル本体とメタ情報をバックエンドへ送信する。

1ファイルあたりの容量上限は5MBとする。

許可するファイル種別は以下とする。

- 画像
- PDF
- テキスト
- Office系ファイル

フロントエンドでは、アップロード前に容量と種別を確認する。

ただし、最終的な判定はバックエンドでも行う。

## 21. ドラッグアンドドロップ方針

カンバン画面では、タスクカードをドラッグアンドドロップしてStatusを変更できる。

ドロップ時に移動先Statusを判定し、タスクステータス変更APIを呼び出す。

API更新に成功した場合、画面上のStatusを確定する。

API更新に失敗した場合、タスクカードを移動前のStatusに戻す。

ドラッグアンドドロップに使用するライブラリは後続工程で検討する。
