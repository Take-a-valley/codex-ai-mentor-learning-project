# 基本設計書

## 1. 基本設計の目的

本書は、要件定義書で整理した内容をもとに、初期MVPであるカンバン方式タスク管理機能のシステム構成、主要機能、画面、認証・認可、データ管理方針を整理する。

後続工程である詳細設計、DB設計、API設計、画面設計、テスト設計へ進むための前提を明確にすることを目的とする。

## 2. 対象範囲

本書では、初期MVPとして以下を対象とする。

- メールログイン
- パスワードリセット
- 複数プロジェクト管理
- プロジェクトメンバー招待
- ロール管理
- カンバン方式のタスク管理
- タスクコメント
- タスク添付ファイル
- 権限制御

以下は本書の対象外とする。

- Wikiによるドキュメント管理
- WBS、ガントチャート
- プロジェクト内チャット
- ステータス列のカスタマイズ
- 外部ストレージ連携
- 本番デプロイ構成の確定

## 3. システム全体構成

本システムは、フロントエンド、バックエンド、データベース、ファイル保存領域、メール送信機能で構成する。

- フロントエンド: Next.js
- バックエンド: Spring Boot
- データベース: MySQL
- ファイル保存: 初期MVPではDB保存
- メール送信: パスワードリセット、プロジェクト招待に使用

フロントエンドは画面表示とユーザー操作を担当する。  

バックエンドは業務ロジック、認証・認可、データ操作、メール送信、ファイル操作を担当する。  

MySQLはユーザー、プロジェクト、プロジェクトメンバー、ロール、タスク、コメント、添付ファイル、招待、パスワードリセット情報を管理する。

開発環境では、MySQL 8系とMailpitをDocker Composeで起動する。

バックエンドはJava 25、Spring Boot 4.1.x系、Maven 3.9.9を前提とする。

フロントエンドはNode.js 24.14.1、npm 11.11.0、Next.js 16.x系または作成時点のlatestを前提とする。

## 4. フロントエンド構成方針

フロントエンドでは、ユーザーが以下の操作を行える画面を提供する。

- ログイン
- パスワードリセット申請
- パスワード再設定
- プロジェクト一覧確認
- プロジェクト作成・編集
- カンバン画面でのタスク確認・操作
- タスク詳細確認
- コメント操作
- 添付ファイル操作
- メンバー管理
- メンバー招待

画面ごとに、ログイン状態とロールに応じた表示制御を行う。  
ただし、最終的な権限制御はバックエンド側でも必ず行う。

## 5. バックエンド構成方針

バックエンドでは、以下の責務を持つ。

- 認証処理
- パスワードリセット処理
- プロジェクト管理処理
- メンバー招待処理
- ロール管理処理
- タスク管理処理
- コメント管理処理
- 添付ファイル管理処理
- メール送信処理
- 権限制御処理

バックエンドは、フロントエンドからのリクエストに対して、認証状態と権限を確認したうえで処理を実行する。

## 6. データ管理方針

MySQLで以下の情報を管理する。

- ユーザー
- プロジェクト
- プロジェクトメンバー
- ロール
- タスク
- タスクコメント
- 添付ファイルメタ情報
- 招待情報
- パスワードリセット情報

添付ファイル本体は初期MVPではMySQLに保存し、DBには、ファイル名、ファイル種別、サイズ、ファイル本体データ、紐づくタスクなどを保存する。

## 7. 認証設計方針

ユーザーはメールアドレスとパスワードでログインする。

パスワードを忘れた場合、パスワードリセット申請を行う。  
システムはリセット用URLをメール送信する。  
リセットURLの有効期限は30分とする。

初期MVPでは、開発環境用の初期データとしてシステム管理者を作成する。  
初期パスワードは要件定義書及び設計書には記載せず、環境変数または初期セットアップ時に設定する。

## 8. 認可設計方針

本システムでは、システム全体ロールとプロジェクト内ロールを分ける。

### システム全体ロール

- システム管理者

システム管理者は、全ユーザー、全プロジェクトを確認できる。  
必要に応じて、システム全体の管理操作を行う。

### プロジェクト内ロール

- プロジェクト管理者
- メンバー
- 閲覧者

プロジェクト管理者は、対象プロジェクト内の管理操作を行う。  
メンバーは、対象プロジェクト内でタスク操作やコメント操作を行う。  
閲覧者は、対象プロジェクト内の情報を閲覧する。

権限のない操作は、画面上で非表示または無効化し、バックエンドでも拒否する。

| メンバー管理 | 可 | 可 | 不可 | 不可 |
| ロール変更 | 可 | 可 | 不可 | 不可 |
| タスク閲覧 | 可 | 可 | 可 | 可 |
| タスク作成 | 可 | 可 | 可 | 不可 |
| タスク編集 | 可 | 可 | 可 | 不可 |
| タスク移動 | 可 | 可 | 可 | 不可 |
| タスク削除 | 可 | 可 | 不可 | 不可 |

## 10. プロジェクト管理設計方針

ユーザーは複数プロジェクトを切り替えて利用できる。

プロジェクト作成はシステム管理者のみ可能とする。

プロジェクト作成直後は、作成したシステム管理者を対象プロジェクトのプロジェクト管理者として登録する。

システム管理者は、作成後に任意のユーザーを対象プロジェクトのプロジェクト管理者へ変更できる。

プロジェクト内の管理操作は、システム管理者または対象プロジェクトのプロジェクト管理者が行う。

システム管理者は、システム全体の管理目的で全プロジェクトを確認できる。

プロジェクト管理者自身のロール変更は、対象プロジェクト内に他のプロジェクト管理者が存在する場合のみ可能とする。

## 11. メンバー招待設計方針

プロジェクト管理者またはシステム管理者は、メールアドレスを指定してユーザーをプロジェクトへ招待できる。

招待URLの有効期限は24時間とする。

未登録ユーザーは、招待URLからユーザー登録後、対象プロジェクトに参加できる。  
招待時の初期ロールはメンバー固定とする。

## 12. タスク管理設計方針

タスクはカンバン方式で管理する。

タスクには以下の情報を持たせる。

- Summary
- Description
- Status
- Assignee
- Due date
- Priority
- Creator
- Created date
- Updated date

Statusは以下の4つとする。

- ToDo
- In Progress
- Review
- Done

Due dateは任意入力とし、入力する場合は未来の日付のみ選択可能とする。

Priorityは以下から選択する。

- Highest
- High
- Medium
- Low
- Lowest

メンバーはタスクの作成、編集、移動ができる。  
タスク削除はシステム管理者またはプロジェクト管理者のみ可能とする。

## 13. コメント管理設計方針

タスクにはコメントを登録できる。

メンバーはコメントを作成できる。  
メンバーは自分のコメントのみ編集できる。  
コメント削除はシステム管理者またはプロジェクト管理者のみ可能とする。  
閲覧者はコメントを閲覧できるが、作成・編集・削除はできない。

## 14. 添付ファイル管理設計方針

タスクにはファイルを添付できる。

添付ファイルは以下の操作に対応する。

- アップロード
- ダウンロード
- 削除

添付ファイル削除は、システム管理者またはプロジェクト管理者のみ可能とする。

添付ファイル本体は初期MVPではMySQLに保存する。
DBには、ファイル名、ファイル種別、サイズ、ファイル本体データ、紐づくタスクなどを保存する。

1ファイルあたりの容量上限は5MBとする。

許可するファイル種別は以下とする。

- 画像
- PDF
- テキスト
- Office系ファイル

## 15. メール送信設計方針

本システムでは以下のメールを送信する。

- パスワードリセットメール
- プロジェクト招待メール

パスワードリセットメールには、30分間有効なリセットURLを含める。

プロジェクト招待メールには、24時間有効な招待URLを含める。

## 16. 画面構成方針

初期MVPでは以下の画面を想定する。

- ログイン画面
- パスワードリセット申請画面
- パスワード再設定画面
- プロジェクト一覧画面
- プロジェクト作成・編集画面
- カンバン画面
- タスク詳細画面
- メンバー管理画面
- メンバー招待画面

各画面では、ログイン状態、参加プロジェクト、ロールに応じて表示・操作できる内容を制御する。

## 17. 主要画面の役割

### ログイン画面

メールアドレスとパスワードを入力してログインする。

### パスワードリセット申請画面

メールアドレスを入力し、パスワードリセットメールを送信する。

### パスワード再設定画面

リセットURLからアクセスし、新しいパスワードを設定する。

### プロジェクト一覧画面

ユーザーが閲覧可能なプロジェクトを表示する。

### カンバン画面

対象プロジェクトのタスクをステータス別に表示する。  
タスクの作成、編集、移動を行う。

### タスク詳細画面

タスクの詳細情報、コメント、添付ファイルを表示する。

### メンバー管理画面

プロジェクト内メンバーとロールを管理する。

### メンバー招待画面

メールアドレスを入力して、プロジェクトへユーザーを招待する。

## 18. API設計への接続方針

API設計では、以下の単位でエンドポイントを整理する。

- 認証
- パスワードリセット
- プロジェクト
- プロジェクトメンバー
- 招待
- タスク
- コメント
- 添付ファイル

各APIでは、認証状態とロールに基づく認可を必ず確認する。

## 19. DB設計への接続方針

DB設計では、以下のデータを中心にテーブル設計を行う。

- ユーザー
- プロジェクト
- プロジェクトメンバー
- ロール
- タスク
- タスクコメント
- 招待
- パスワードリセット
- ファイル名
- ファイル種別
- ファイルサイズ
- ファイル本体データ

特に、システム全体ロールとプロジェクト内ロールを混同しないように設計する。

## 20. 非機能設計方針

初期MVPでは、以下を重視する。

- 権限のない操作を実行できないこと
- 管理者が0人になる状態を防ぐこと
- 添付ファイルの容量と種別を制限すること
- パスワードリセットURLと招待URLに有効期限を設けること
- 将来の機能追加を考慮し、機能ごとの責務を分けること
- 初学者が理解しやすい構成にすること

## 21. 将来拡張方針

将来的に以下の機能を追加する。

- Wikiによるドキュメント管理
- WBS、ガントチャート
- プロジェクト内チャット
- ステータス列のカスタマイズ
- 外部ストレージ連携
- 通知機能

初期MVPでは、これらを直接実装しない。  
ただし、将来追加できるように、機能ごとの責務を分けて設計する。
