# バックエンド設計書

## 1. 本書の目的

本書は、タスク管理機能におけるバックエンドの基本設計を整理することを目的とする。

要件定義書、基本設計書、DB設計書、API設計書、画面設計書、認証・認可設計書、フロントエンド設計書をもとに、Spring Bootで構築するバックエンドの構成方針、責務分担、認証・認可、トランザクション、入力チェック、エラー制御、データ操作方針を定義する。

## 2. 対象範囲

本書では、初期MVPであるカンバン方式タスク管理機能のバックエンドを対象とする。

対象機能は以下とする。

- 認証
- ログアウト
- ログインユーザー情報取得
- パスワードリセット
- プロジェクト管理
- プロジェクトメンバー管理
- メール招待
- カンバン表示
- タスク管理
- コメント管理
- 添付ファイル管理
- 認可チェック
- 論理削除制御

以下は初期MVPでは対象外とする。

- Wiki機能
- WBS、ガントチャート機能
- プロジェクト内チャット機能
- 通知機能
- 外部ストレージ連携
- ステータス列のカスタマイズ

## 4. バックエンドの責務

バックエンドは以下を担当する。

- APIリクエストの受付
- 認証状態の確認
- 認可チェック
- 入力値の検証
- 業務ルールの実行
- DBへの登録、更新、取得、論理削除
- トランザクション制御
- セッション管理
- CSRF対策
- パスワードの安全な管理
- 招待URL、パスワードリセットURLのトークン管理
- エラーレスポンスの返却
- 添付ファイルの保存、取得、削除制御

画面表示や画面上の操作制御はフロントエンドが担当する。

ただし、認証・認可・業務ルールの最終判定は必ずバックエンドで行う。

## 5. 全体構成方針

バックエンドは、役割ごとに層を分けて構成する。

| 層 | 役割 |
|---|---|
| Controller | HTTPリクエストを受け取り、レスポンスを返す |
| Service | 業務処理、認可チェック、トランザクション制御を行う |
| Repository | DBアクセスを行う |
| Entity | DBテーブルに対応するデータ構造を表す |
| DTO | APIリクエスト、レスポンス用のデータ構造を表す |
| Security | 認証、認可、CSRF、セッション制御を扱う |
| Exception | 業務エラー、入力エラー、システムエラーを扱う |

Controllerには業務判断を持たせない。

業務ルールはServiceに集約する。

RepositoryはDB操作に責務を限定する。

## 6. 機能構成方針

バックエンドは以下の機能単位で構成する。

| 機能 | 内容 |
|---|---|
| Auth | ログイン、ログアウト、ログインユーザー取得 |
| PasswordReset | パスワードリセット申請、再設定 |
| Project | プロジェクト作成、取得、更新、削除 |
| ProjectMember | メンバー一覧、ロール変更、メンバー削除 |
| Invitation | 招待作成、招待取得、招待承認 |
| Task | タスク作成、取得、更新、ステータス変更、削除 |
| Comment | コメント作成、更新、削除 |
| Attachment | 添付ファイル追加、取得、削除 |
| Authorization | システムロール、プロジェクトロールに基づく認可判定 |

## 7. Controller設計方針

Controllerは、API設計書で定義したエンドポイントに対応する。

Controllerの責務は以下とする。

- URL、HTTPメソッド、リクエスト形式の受け取り
- リクエストDTOへの変換
- 入力形式の基本チェック呼び出し
- Serviceの呼び出し
- レスポンスDTOの返却
- HTTPステータスの返却

Controllerでは以下を行わない。

- 複雑な業務判断
- DBアクセス
- ロール判定の詳細実装
- トランザクション境界の管理

## 8. Service設計方針

Serviceは業務処理の中心とする。

Serviceの責務は以下とする。

- 業務ルールの実行
- 認可チェック
- DB更新を伴う処理のトランザクション制御
- 論理削除済みデータの扱い
- 関連データの整合性確認
- Repositoryの呼び出し
- 業務エラーの判定

プロジェクト作成、タスク作成、ロール変更、プロジェクト削除など、複数テーブルを更新する処理はServiceで一貫して管理する。

## 9. Repository設計方針

RepositoryはDBアクセスを担当する。

Repositoryの責務は以下とする。

- Entityの取得
- Entityの登録
- Entityの更新
- 論理削除状態の更新
- 検索条件に基づく一覧取得

Repositoryでは業務ルールを判断しない。

論理削除済みデータは、通常の一覧取得や詳細取得では取得対象外とする。

履歴確認が必要なデータについては、専用の取得条件を設ける。

## 10. 認証設計方針

認証方式はセッション方式とする。

ログイン成功時、バックエンドはサーバー側でセッションを作成し、ブラウザへCookieを返却する。

以降の認証必須APIでは、Cookieに紐づくセッションからログインユーザーを判定する。

未ログイン状態で認証必須APIへアクセスした場合は、401 Unauthorizedを返却する。

ログアウト時はセッションを無効化する。

## 11. CSRF対策方針

セッション方式を採用するため、状態変更APIではCSRF対策を行う。

対象となるHTTPメソッドは以下とする。

- POST
- PUT
- PATCH
- DELETE

フロントエンドはCSRFトークンを取得し、状態変更APIのリクエスト時に送信する。

CSRFトークンが不正または不足している場合、バックエンドは403 Forbiddenを返却する。

CSRFトークンの取得方法、ヘッダー名、Cookie名は後続工程で具体化する。

## 12. 認可設計方針

バックエンドでは、すべての認証必須APIで認可チェックを行う。

認可判定では、以下を使用する。

- システムロール
- 対象プロジェクトの参加状態
- 対象プロジェクト内ロール
- 操作対象データの所有者
- 論理削除状態

システム管理者は、全プロジェクトに対する管理操作を行える。

通常ユーザーは、参加しているプロジェクト内で付与されたプロジェクトロールに応じて操作できる。

プロジェクト管理者は、対象プロジェクトに対して管理操作を行える。

メンバーは、タスク作成、タスク編集、タスク移動、自分のコメント編集などを行える。

閲覧者は、参照のみ可能とする。

## 13. プロジェクト作成処理方針

プロジェクト作成はシステム管理者のみ可能とする。

プロジェクト作成時は、以下を同一トランザクションで実行する。

1. ログインユーザーがシステム管理者であることを確認する
2. プロジェクトを登録する
3. 作成したシステム管理者をProjectMemberに登録する
4. project_role_idにPROJECT_ADMINを設定する

作成直後のプロジェクトには、必ず1人以上のPROJECT_ADMINが存在する。

システム管理者は、プロジェクト作成後に任意のユーザーをPROJECT_ADMINへ変更できる。

## 14. プロジェクト更新・削除処理方針

プロジェクト更新は、システム管理者または対象プロジェクトのプロジェクト管理者のみ可能とする。

プロジェクト削除は論理削除とする。

プロジェクト削除時は、対象プロジェクトに紐づく以下のデータも論理削除扱いとする。

- タスク
- コメント
- 添付ファイル
- プロジェクトメンバー

招待情報は履歴として残す。

プロジェクト削除処理は、関連データの更新を含むため同一トランザクションで実行する。

## 15. メンバー管理処理方針

メンバー招待、ロール変更、メンバー削除は、システム管理者または対象プロジェクトのプロジェクト管理者のみ可能とする。

招待時の初期ロールはMEMBER固定とする。

ロール変更では、PROJECT_ADMIN、MEMBER、VIEWERのいずれかを設定できる。

ロール変更またはメンバー削除により、対象プロジェクトのPROJECT_ADMINが0人になる操作は禁止する。

操作対象がシステム管理者であっても、対象プロジェクトのPROJECT_ADMINとして登録されている場合は、PROJECT_ADMINが0人にならないことを確認する。

## 16. 招待処理方針

管理者はメールアドレスを指定してユーザーを招待する。

招待作成時は、招待トークンを発行する。

招待URLの有効期限は24時間とする。

未登録ユーザーは、招待URLからユーザー登録後、対象プロジェクトへ参加できる。

登録済みユーザーは、ログイン後に招待を承認し、対象プロジェクトへ参加できる。

招待情報は履歴として残す。

招待ステータスを管理し、承認済み、期限切れ、無効などの状態を判定できるようにする。

## 17. タスク管理処理方針

タスクはプロジェクトに紐づく。

タスクのStatusは初期MVPでは以下の固定値とする。

| 保存値 | 表示名 |
|---|---|
| TODO | ToDo |
| IN_PROGRESS | In Progress |
| REVIEW | Review |
| DONE | Done |

タスクのPriorityは初期MVPでは以下の固定値とする。

| 保存値 | 表示名 |
|---|---|
| HIGHEST | Highest |
| HIGH | High |
| MEDIUM | Medium |
| LOW | Low |
| LOWEST | Lowest |

タスク作成、編集、ステータス変更は、システム管理者、対象プロジェクトのプロジェクト管理者、メンバーが実行できる。

タスク削除は、システム管理者または対象プロジェクトのプロジェクト管理者のみ可能とする。

タスク削除は論理削除とする。

ドラッグアンドドロップによるStatus変更では、移動先Statusを受け取り、対象タスクのStatusを更新する。

## 18. コメント処理方針

コメントはタスクに紐づく。

コメント作成は、システム管理者、対象プロジェクトのプロジェクト管理者、メンバーが実行できる。

コメント編集は、コメント投稿者本人のみ可能とする。

コメント削除は、システム管理者または対象プロジェクトのプロジェクト管理者のみ可能とする。

コメント削除は論理削除とする。

ユーザーが削除済みの場合でも、作成済みコメントは削除しない。

削除済みユーザーの表示名は「削除済ユーザー」として扱う。

## 19. 添付ファイル処理方針

添付ファイルはタスクに紐づく。

添付ファイルの本体はDBに保存する。

添付ファイル追加時は、ファイル本体とメタ情報を保存する。

保存するメタ情報は以下を想定する。

- ファイル名
- MIMEタイプ
- ファイルサイズ
- 登録者
- 登録日時

1ファイルあたりの容量上限は5MBとする。

許可するファイル種別は以下とする。

- 画像
- PDF
- テキスト
- Office系ファイル

添付ファイル削除は論理削除とする。

添付ファイル取得時は、対象添付ファイルが指定されたタスクおよびプロジェクトに紐づいていることを確認する。

論理削除されている添付ファイルは返却しない。

## 20. パスワードリセット処理方針

パスワードリセット申請では、メールアドレスを受け取り、対象ユーザーが存在する場合にリセットトークンを発行する。

対象メールアドレスの存在有無に関わらず、フロントエンドへは同じ完了レスポンスを返す。

パスワードリセットURLの有効期限は30分とする。

パスワード再設定時は、トークンの存在、有効期限、使用済み状態を確認する。

パスワード再設定完了後、トークンは使用済みとして扱う。

パスワードリセット情報は履歴として残す。

## 21. 入力チェック方針

バックエンドでは、すべてのリクエストに対して入力チェックを行う。

主なチェック項目は以下とする。

| 対象 | チェック内容 |
|---|---|
| メールアドレス | 必須、形式 |
| パスワード | 必須 |
| プロジェクト名 | 必須 |
| Summary | 必須 |
| Due date | 日付形式 |
| Priority | 許可された値 |
| Status | 許可された値 |
| コメント本文 | 必須 |
| 添付ファイル | サイズ上限、MIMEタイプ |

フロントエンド側のチェックは補助であり、バックエンドで必ず再チェックする。

## 22. エラー制御方針

バックエンドは、エラー内容に応じて適切なHTTPステータスを返却する。

| HTTPステータス | 用途 |
|---|---|
| 400 | 入力値不正 |
| 401 | 未認証 |
| 403 | 権限不足、CSRF不正 |
| 404 | 対象データなし |
| 409 | 業務制約違反 |
| 413 | ファイルサイズ超過 |
| 415 | 許可されていないファイル種別 |
| 500 | サーバー内部エラー |

エラーレスポンスは、API設計書の共通エラー形式に合わせる。

## 23. トランザクション方針

複数テーブルを更新する処理は、同一トランザクションで実行する。

対象例は以下とする。

- プロジェクト作成
- プロジェクト削除
- メンバー招待承認
- ロール変更
- タスク削除
- 添付ファイル追加
- パスワード再設定

処理途中でエラーが発生した場合は、更新内容をロールバックする。

## 24. 論理削除方針

各データの削除は物理削除ではなく論理削除とする。

論理削除では、削除フラグまたは削除日時を更新する。

通常の取得APIでは、論理削除済みデータを返却しない。

ユーザー削除時は、作成済みタスクやコメントは削除しない。

削除済みユーザーの表示名は「削除済ユーザー」とする。

プロジェクト削除時は、タスク、コメント、添付ファイルも論理削除扱いとする。

招待情報、パスワードリセット情報は履歴として残す。

## 25. セキュリティ方針

バックエンドでは以下を考慮する。

- パスワードはハッシュ化して保存する
- セッションCookieには安全な属性を設定する
- CSRF対策を行う
- 認可チェックをすべての対象APIで実施する
- 添付ファイルのサイズと種別を検証する
- エラーメッセージで機密情報を返さない
- パスワードリセット申請時にユーザー存在有無を推測させない
- トークンは推測困難な値とする
- トークンは有効期限と使用済み状態を管理する

## 26. ログ出力方針

バックエンドでは、運用上必要なログを出力する。

対象例は以下とする。

- ログイン成功、失敗
- ログアウト
- プロジェクト作成、更新、削除
- ロール変更
- 招待作成、承認
- パスワードリセット申請、完了
- 業務エラー
- システムエラー

ログにはパスワード、トークン、セッションIDなどの機密情報を出力しない。

## 27. テスト方針

バックエンド実装時は、以下の観点でテストを行う。

- 正常系API
- 入力チェック
- 認証必須APIの未ログインアクセス
- ロール別認可チェック
- PROJECT_ADMINが0人になる操作の禁止
- プロジェクト作成時のPROJECT_ADMIN自動登録
- 論理削除後の取得制御
- 添付ファイルのサイズ、種別チェック
- パスワードリセットトークンの有効期限
- 招待トークンの有効期限
