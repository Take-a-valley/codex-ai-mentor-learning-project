# テスト方針書

## 1. 本書の目的

本書は、タスク管理機能の初期MVPにおけるテスト方針を整理することを目的とする。

要件定義書、基本設計書、DB設計書、API設計書、画面設計書、認証・認可設計書、フロントエンド設計書、バックエンド設計書をもとに、どの範囲を、どの観点で、どの段階で確認するかを定義する。

## 2. 対象範囲

本書では、初期MVPであるカンバン方式タスク管理機能を対象とする。

対象機能は以下とする。

- 認証
- ログアウト
- ログインユーザー情報取得
- パスワードリセット
- プロジェクト管理
- プロジェクトメンバー管理
- メール招待
- カンバン表示
- タスク管理
- コメント管理
- 添付ファイル管理
- 認可チェック
- 論理削除制御

以下は初期MVPでは対象外とする。

- Wiki機能
- WBS、ガントチャート機能
- プロジェクト内チャット機能
- 通知機能
- 外部ストレージ連携
- ステータス列のカスタマイズ

## 3. テストの基本方針

テストでは、以下を重視する。

- 要件どおりに機能が動作すること
- 認証・認可が正しく制御されること
- 不正な入力や不正な操作を防げること
- 論理削除後のデータが通常操作に表示されないこと
- プロジェクト管理者が0人にならないこと
- 添付ファイルの容量と種別が制限されること
- 招待URLとパスワードリセットURLの有効期限が守られること
- フロントエンド、バックエンド、DBの連携が成立すること

初期MVPでは、すべてを自動化するのではなく、重要度の高い箇所から優先して確認する。

## 4. テスト種別

初期MVPでは、以下のテストを実施する。

| テスト種別 | 目的 |
|---|---|
| 単体テスト | API、画面、Serviceなど、個別の処理や部品が正しく動くことを確認する |
| 結合テスト | フロントエンド、バックエンド、DB、認証・認可、業務ルールの連携を確認する |
| 回帰テスト | 修正後に既存機能が壊れていないことを確認する |

本書では、APIテストと画面テストは単体テストに含める。

権限テスト、認証テスト、CSRFテスト、パスワードリセットテスト、招待テスト、プロジェクト管理テスト、メンバー管理テスト、タスク管理テスト、コメントテスト、添付ファイルテスト、論理削除テスト、入力チェックテスト、エラーレスポンステストは結合テストに含める。

## 5. 単体テスト方針

単体テストでは、個別の処理、API、画面が設計どおりに動作することを確認する。

単体テストの対象は以下とする。

- バックエンド単体テスト
- APIテスト
- 画面テスト

## 6. バックエンド単体テスト方針

バックエンド単体テストでは、主にService層を対象とする。

対象例は以下とする。

- 認可判定
- プロジェクト作成時のPROJECT_ADMIN自動登録
- PROJECT_ADMINが0人になる操作の禁止
- 招待トークンの有効期限判定
- パスワードリセットトークンの有効期限判定
- タスクStatus更新
- 論理削除判定
- 添付ファイルのサイズ判定
- 添付ファイルの種別判定

Controller、Repositoryの単体テスト範囲は、実装方針決定後に具体化する。

## 7. APIテスト方針

APIテストでは、API設計書に定義した各APIについて、API単位でリクエストとレスポンスを確認する。

確認観点は以下とする。

- 正しいHTTPメソッドで呼び出せること
- 正しいパスで呼び出せること
- 必須項目が不足した場合に400を返すこと
- 未ログイン時に401を返すこと
- 代表的な権限不足時に403を返すこと
- 対象データが存在しない場合に404を返すこと
- 業務制約違反時に409を返すこと
- ファイルサイズ超過時に413を返すこと
- 許可されていないファイル種別の場合に415を返すこと
- 正常時にAPI設計書どおりのJSONを返すこと

削除系APIでは、物理削除ではなく論理削除されることを確認する。

## 8. 画面テスト方針

画面テストでは、画面設計書とフロントエンド設計書に基づいて、画面単位で表示と操作を確認する。

確認観点は以下とする。

- 画面が正しく表示されること
- 入力項目が表示されること
- 必須入力チェックが行われること
- 操作ボタンが期待どおり表示されること
- 画面単位で、ロールに応じたボタンやリンクの表示制御が行われること
- APIエラー時に適切なメッセージが表示されること
- ローディング中に二重送信を防止できること
- 空状態が表示されること
- 確認ダイアログが表示されること
- 画面遷移が正しく行われること

画面テストでは、画面単体の表示・入力・操作確認を中心とする。

フロントエンド、バックエンド、DBを組み合わせた一連の業務シナリオ確認は結合テストで扱う。

## 9. 結合テスト方針

結合テストでは、フロントエンド、バックエンド、DB、認証・認可、業務ルールを組み合わせて確認する。

結合テストの対象は以下とする。

- 権限テスト
- 認証テスト
- CSRFテスト
- パスワードリセットテスト
- 招待テスト
- プロジェクト管理テスト
- メンバー管理テスト
- タスク管理テスト
- コメントテスト
- 添付ファイルテスト
- 論理削除テスト
- 入力チェックテスト
- エラーレスポンステスト

## 10. 結合テストの主なシナリオ

主な確認シナリオは以下とする。

- ログインしてプロジェクト一覧を表示する
- システム管理者がプロジェクトを作成する
- プロジェクト作成直後に作成者がPROJECT_ADMINになる
- システム管理者がメンバーを招待する
- 招待されたユーザーがプロジェクトに参加する
- プロジェクト管理者がメンバーのロールを変更する
- メンバーがタスクを作成する
- メンバーがタスクを編集する
- ドラッグアンドドロップでStatusを変更する
- コメントを作成、編集、削除する
- 添付ファイルを追加、取得、削除する
- プロジェクトを論理削除する

## 11. 権限テスト方針

権限テストでは、以下のロールごとに操作可否を確認する。

- システム管理者
- 対象プロジェクトのプロジェクト管理者
- メンバー
- 閲覧者
- 未ログインユーザー
- 対象プロジェクトに参加していないユーザー

主な確認観点は以下とする。

| 操作 | 確認内容 |
|---|---|
| プロジェクト作成 | システム管理者のみ実行できること |
| プロジェクト編集 | システム管理者または対象プロジェクトのプロジェクト管理者のみ実行できること |
| プロジェクト削除 | システム管理者または対象プロジェクトのプロジェクト管理者のみ実行できること |
| メンバー招待 | システム管理者または対象プロジェクトのプロジェクト管理者のみ実行できること |
| ロール変更 | システム管理者または対象プロジェクトのプロジェクト管理者のみ実行できること |
| タスク作成 | システム管理者、対象プロジェクトのプロジェクト管理者、メンバーが実行できること |
| タスク編集 | システム管理者、対象プロジェクトのプロジェクト管理者、メンバーが実行できること |
| タスク削除 | システム管理者または対象プロジェクトのプロジェクト管理者のみ実行できること |
| コメント作成 | システム管理者、対象プロジェクトのプロジェクト管理者、メンバーが実行できること |
| コメント編集 | コメント投稿者本人のみ実行できること |
| コメント削除 | システム管理者または対象プロジェクトのプロジェクト管理者のみ実行できること |
| 添付ファイル追加 | システム管理者、対象プロジェクトのプロジェクト管理者、メンバーが実行できること |
| 添付ファイル削除 | システム管理者または対象プロジェクトのプロジェクト管理者のみ実行できること |

## 12. 認証テスト方針

認証では以下を確認する。

- 正しいメールアドレスとパスワードでログインできること
- 誤った認証情報ではログインできないこと
- ログイン成功後にセッションが作成されること
- 認証必須APIでセッションが確認されること
- 未ログイン時に認証必須APIへアクセスすると401になること
- ログアウト後にセッションが無効化されること
- ログアウト後に認証必須APIへアクセスできないこと

## 13. CSRFテスト方針

CSRF対策では以下を確認する。

- 状態変更APIでCSRFトークンが必要であること
- CSRFトークンがない場合に403になること
- 不正なCSRFトークンの場合に403になること
- 正しいCSRFトークンの場合に状態変更APIを実行できること

対象となるHTTPメソッドは以下とする。

- POST
- PUT
- PATCH
- DELETE

## 14. パスワードリセットテスト方針

パスワードリセットでは以下を確認する。

- パスワードリセット申請ができること
- 登録済みメールアドレスの場合にリセットトークンが発行されること
- 未登録メールアドレスの場合でも同じ完了レスポンスになること
- リセットURLの有効期限が30分であること
- 期限切れトークンでは再設定できないこと
- 使用済みトークンは再利用できないこと
- パスワード再設定後に新しいパスワードでログインできること
- パスワードリセット情報が履歴として残ること

## 15. 招待テスト方針

招待では以下を確認する。

- システム管理者または対象プロジェクトのプロジェクト管理者が招待を作成できること
- 招待時の初期ロールがMEMBERであること
- 招待URLの有効期限が24時間であること
- 未登録ユーザーが招待URLから登録して参加できること
- 登録済みユーザーがログイン後に招待を承認できること
- 招待先メールアドレスとログインユーザーのメールアドレスが一致しない場合は承認できないこと
- 承認済み招待は再利用できないこと
- 期限切れ招待は承認できないこと
- 招待情報が履歴として残ること

## 16. プロジェクト管理テスト方針

プロジェクト管理では以下を確認する。

- システム管理者のみプロジェクトを作成できること
- プロジェクト作成直後に作成者がPROJECT_ADMINとして登録されること
- システム管理者が全プロジェクトを確認できること
- 通常ユーザーが参加済みプロジェクトのみ確認できること
- システム管理者または対象プロジェクトのプロジェクト管理者のみプロジェクトを編集できること
- プロジェクト削除が論理削除であること
- プロジェクト削除後、通常の一覧に表示されないこと
- プロジェクト削除時に紐づくタスク、コメント、添付ファイルも論理削除扱いになること

## 17. メンバー管理テスト方針

メンバー管理では以下を確認する。

- メンバー一覧を取得できること
- ロール変更ができること
- MEMBER、VIEWER、PROJECT_ADMINへ変更できること
- PROJECT_ADMINが0人になるロール変更が禁止されること
- PROJECT_ADMINが0人になるメンバー削除が禁止されること
- メンバー削除が論理削除であること
- 論理削除されたメンバーが通常のメンバー一覧に表示されないこと

## 18. タスク管理テスト方針

タスク管理では以下を確認する。

- タスクを作成できること
- タスク作成時の初期StatusがTODOであること
- StatusにTODO、IN_PROGRESS、REVIEW、DONEのみ設定できること
- PriorityにHIGHEST、HIGH、MEDIUM、LOW、LOWESTのみ設定できること
- タスク一覧がStatus別に取得できること
- タスク詳細を取得できること
- タスクを編集できること
- タスクのStatusを変更できること
- ドラッグアンドドロップ後のStatus変更が反映されること
- タスク削除が論理削除であること
- 論理削除されたタスクが通常の一覧に表示されないこと

## 19. コメントテスト方針

コメントでは以下を確認する。

- コメントを作成できること
- コメント本文が必須であること
- コメント投稿者本人のみコメントを編集できること
- 他ユーザーのコメントを編集できないこと
- システム管理者または対象プロジェクトのプロジェクト管理者がコメントを削除できること
- コメント削除が論理削除であること
- 論理削除されたコメントが通常表示されないこと
- 削除済みユーザーのコメント投稿者名が「削除済ユーザー」と表示されること

## 20. 添付ファイルテスト方針

添付ファイルでは以下を確認する。

- 許可されたファイル種別を追加できること
- 1ファイルあたり5MB以下のファイルを追加できること
- 5MBを超えるファイルは追加できないこと
- 許可されていないファイル種別は追加できないこと
- 添付ファイルを取得できること
- 対象添付ファイルが指定されたタスクおよびプロジェクトに紐づくことを確認すること
- 添付ファイル削除が論理削除であること
- 論理削除された添付ファイルを取得できないこと

## 21. 論理削除テスト方針

論理削除では以下を確認する。

- 削除操作で物理削除されないこと
- deleted_atまたは削除状態が更新されること
- 論理削除済みデータが通常の一覧に表示されないこと
- 論理削除済みデータが通常の詳細取得で取得できないこと
- ユーザー削除時、作成済みタスクやコメントは削除されないこと
- 削除済みユーザーの表示名が「削除済ユーザー」になること
- 招待情報とパスワードリセット情報が履歴として残ること

## 22. 入力チェックテスト方針

入力チェックでは以下を確認する。

| 対象 | 確認内容 |
|---|---|
| メールアドレス | 必須、形式 |
| パスワード | 必須 |
| プロジェクト名 | 必須 |
| Summary | 必須 |
| Due date | 日付形式 |
| Priority | 許可された値 |
| Status | 許可された値 |
| コメント本文 | 必須 |
| 添付ファイル | サイズ上限、ファイル種別 |

フロントエンドの入力チェックだけでなく、バックエンドでも同じ制約が確認されることをテストする。

## 23. エラーレスポンステスト方針

エラーレスポンスでは以下を確認する。

| HTTPステータス | 確認内容 |
|---|---|
| 400 | 入力値不正時に返却されること |
| 401 | 未認証時に返却されること |
| 403 | 権限不足またはCSRF不正時に返却されること |
| 404 | 対象データが存在しない場合に返却されること |
| 409 | 業務制約違反時に返却されること |
| 413 | ファイルサイズ超過時に返却されること |
| 415 | 許可されていないファイル種別の場合に返却されること |
| 500 | サーバー内部エラー時に返却されること |

エラーレスポンスのJSON構造はAPI設計書に従う。

## 24. 回帰テスト方針

回帰テストでは、修正後に既存機能が壊れていないことを確認する。

対象例は以下とする。

- ログインできること
- プロジェクト一覧が表示できること
- カンバン画面が表示できること
- タスク作成、編集、移動ができること
- コメント操作ができること
- 添付ファイル操作ができること
- ロール別の表示制御が維持されていること
- 主要なAPIが正常に動作すること

## 25. テストデータ方針

テストでは、以下のユーザーとプロジェクトを用意する。

| データ | 内容 |
|---|---|
| システム管理者 | 全体管理操作を確認するユーザー |
| プロジェクト管理者 | 対象プロジェクトの管理操作を確認するユーザー |
| メンバー | タスク操作、コメント操作を確認するユーザー |
| 閲覧者 | 参照のみ可能であることを確認するユーザー |
| 未参加ユーザー | 対象プロジェクトへアクセスできないことを確認するユーザー |
| 削除済みユーザー | 表示名が「削除済ユーザー」になることを確認するユーザー |

プロジェクトは、参加済みプロジェクト、未参加プロジェクト、論理削除済みプロジェクトを用意する。

## 26. テスト実施順序

テストは以下の順序で進める。

1. バックエンド単体テスト
2. APIテスト
3. 画面テスト
4. 結合テスト
5. 回帰テスト

認証・認可・論理削除は影響範囲が広いため、結合テストで重点的に確認する。

## 27. 合格基準

テストの合格基準は以下とする。

- 主要な正常系操作がすべて成功すること
- 認証・認可が設計どおりに制御されること
- 重要な異常系で適切なHTTPステータスとエラーが返ること
- 論理削除済みデータが通常操作に表示されないこと
- PROJECT_ADMINが0人になる操作が禁止されること
- 招待URLとパスワードリセットURLの有効期限が守られること
- 添付ファイルの容量と種別制限が守られること
- 既存機能に重大な回帰がないこと
