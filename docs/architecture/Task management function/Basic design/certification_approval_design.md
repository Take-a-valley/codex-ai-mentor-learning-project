# 認証・認可設計書

## 1. 本書の目的

本書は、タスク管理機能における認証、認可、セッション管理、Cookie管理、CSRF対策、ロール判定方針を整理することを目的とする。

要件定義書、基本設計書、DB設計書、API設計書、画面設計書をもとに、ログイン後のユーザー識別、API利用可否、画面表示制御、権限不足時の扱いを明確にする。

後続工程であるフロントエンド詳細設計、バックエンド詳細設計、実装、テスト設計へ進むための前提を定義する。

## 2. 対象範囲

本書では、初期MVPであるタスク管理機能に必要な認証・認可を対象とする。

対象は以下とする。

- メールアドレスとパスワードによるログイン
- ログアウト
- ログインユーザー情報取得
- セッション方式によるログイン状態管理
- Cookieによるセッション識別
- CSRF対策
- パスワードリセット
- プロジェクト招待URL経由の参加
- システムロールによる認可
- プロジェクト内ロールによる認可
- 画面表示制御
- API認可チェック

以下は初期MVPでは対象外とする。

- OAuthログイン
- SNSログイン
- SSO
- 多要素認証
- JWT方式
- APIキー認証
- 外部認証基盤連携

## 3. 認証方式

本システムでは、認証方式としてセッション方式を採用する。

ユーザーがログインに成功した場合、バックエンドはサーバー側にセッションを作成する。

ブラウザには、セッションを識別するためのCookieを保存する。

ログイン後のAPIリクエストでは、ブラウザがCookieを送信し、バックエンドはCookieに紐づくセッション情報からログイン済みユーザーを判定する。

## 4. セッション方式を採用する理由

初期MVPでは、フロントエンドがブラウザアプリケーションであり、バックエンドと同一システムとして連携する想定である。

そのため、ブラウザと相性がよく、サーバー側でログイン状態を管理しやすいセッション方式を採用する。

セッション方式では、ログアウト時にサーバー側のセッションを破棄できるため、ログイン状態の無効化を制御しやすい。

また、学習目的として、Cookie、セッション、CSRF対策、サーバー側認証管理を理解しやすい。

## 5. 認証に利用するAPI

| API | 用途 | 認証 |
|---|---|---|
| POST /api/auth/login | ログイン | 不要 |
| POST /api/auth/logout | ログアウト | 必要 |
| GET /api/auth/me | ログインユーザー取得 | 必要 |
| POST /api/password-reset/request | パスワードリセット申請 | 不要 |
| POST /api/password-reset/confirm | パスワード再設定 | 不要 |
| GET /api/invitations/{token} | 招待情報取得 | 不要 |
| POST /api/invitations/{token}/register | 未登録ユーザー登録・招待承認 | 不要 |
| POST /api/invitations/{token}/accept | 登録済みユーザー招待承認 | 必要 |

## 6. ログイン処理

ユーザーはログイン画面でメールアドレスとパスワードを入力する。

バックエンドは、メールアドレスに一致するユーザーを検索し、パスワードが正しいか検証する。

認証に成功した場合、バックエンドはセッションを作成し、レスポンスでセッションCookieを設定する。

認証に失敗した場合、ログイン失敗として扱い、メールアドレスまたはパスワードが正しくない旨を返す。

## 7. ログアウト処理

ログアウトAPIが呼び出された場合、バックエンドは現在のセッションを破棄する。

セッション破棄後、同じCookieを使ったAPIリクエストは未認証として扱う。

フロントエンドはログアウト成功後、ログイン画面へ遷移する。

## 8. ログインユーザー取得

フロントエンドは、認証後の画面表示時に `GET /api/auth/me` を呼び出し、現在ログインしているユーザー情報を取得する。

取得する情報は以下とする。

| 項目 | 内容 |
|---|---|
| userId | ユーザーID |
| email | メールアドレス |
| displayName | 表示名 |
| systemRole | システムロール |

この情報は、画面表示制御やログイン状態判定に使用する。

ただし、最終的な認可判定はバックエンドAPI側で行う。

## 9. Cookie設計

セッションCookieは、ログイン成功時にバックエンドが発行する。

Cookieにはセッション識別子のみを保持し、ユーザー情報やロール情報は直接保存しない。

Cookieの基本方針は以下とする。

| 項目 | 方針 |
|---|---|
| HttpOnly | 有効 |
| Secure | 本番環境では有効 |
| SameSite | Laxを基本とする |
| Path | / |
| 有効期限 | セッション有効期限に従う |

HttpOnlyを有効にすることで、JavaScriptからCookieを直接参照できないようにする。

SecureはHTTPS通信時のみCookieを送信するため、本番環境では有効にする。

SameSiteは、外部サイトからの意図しないリクエストを抑制するために設定する。

## 10. セッション有効期限

初期MVPでは、セッション有効期限を設ける。

有効期限は、最終アクセスから一定時間で期限切れとする方式を基本とする。

セッション有効期限は、最終アクセスから120分とする。

| 項目 | 値 |
|---|---|
| セッション有効期限 | 最終アクセスから120分 |
| ログアウト時 | 即時無効化 |
| ブラウザ終了時 | Cookie設定に従う |

セッション有効期限が切れた場合、バックエンドは未認証として扱う。

フロントエンドは401エラーを受け取った場合、ログイン画面へ誘導する。

## 11. CSRF対策

本システムでは、セッションCookieを利用するため、CSRF対策を行う。

CSRF対策として、CSRFトークン方式を採用する。

フロントエンドは、状態変更を行うAPIリクエスト時にCSRFトークンを送信する。

バックエンドは、リクエストに含まれるCSRFトークンを検証し、不正な場合はリクエストを拒否する。

CSRF対策が必要なAPIは、主に以下のような状態変更APIである。

- POST
- PUT
- PATCH
- DELETE

GET APIはデータ取得のみとし、状態変更を行わない。

CSRFトークンは、ログイン画面表示時またはアプリ初期表示時にバックエンドから取得する。

フロントエンドは取得したCSRFトークンを、状態変更APIのリクエストヘッダーに設定する。

CSRFトークンの生成と検証はSpring Securityで行う。

フロントエンドがCSRFトークンを取得できるように、CSRFトークン取得APIを用意する。

## 12. パスワード管理

パスワードは平文で保存しない。

バックエンドでは、パスワードをハッシュ化してDBに保存する。

ログイン時は、入力されたパスワードと保存済みハッシュを比較する。

パスワード再設定時は、新しいパスワードをハッシュ化して保存する。

パスワードの具体的なハッシュ方式は、Spring Securityで利用可能な安全な方式を採用する。

## 13. パスワードリセット

ユーザーがパスワードを忘れた場合、パスワードリセット申請を行う。

申請時に入力されたメールアドレスが存在する場合、バックエンドはパスワードリセット用トークンを作成し、リセットURLをメール送信する。

メールアドレスが存在しない場合でも、セキュリティ上、画面には同じ完了メッセージを表示する。

パスワードリセットURLの有効期限は30分とする。

パスワードリセットトークンは、以下の状態を持つ。

| status | 内容 |
|---|---|
| ACTIVE | 使用可能 |
| USED | 使用済み |
| EXPIRED | 期限切れ |

使用済みまたは期限切れのトークンは利用できない。

## 14. 招待URLによる参加

プロジェクト管理者またはシステム管理者は、メールアドレスを指定してユーザーをプロジェクトへ招待できる。

招待URLの有効期限は24時間とする。

招待時の初期ロールはMEMBER固定とする。

招待情報は以下の状態を持つ。

| status | 内容 |
|---|---|
| PENDING | 招待中 |
| ACCEPTED | 承認済み |
| EXPIRED | 期限切れ |
| REVOKED | 無効化済み |

未登録ユーザーは、招待URLからユーザー登録後、対象プロジェクトに参加する。

登録済みユーザーは、ログイン後に招待を承認して対象プロジェクトに参加する。

登録済みユーザーが招待を承認する場合、招待先メールアドレスとログインユーザーのメールアドレスが一致する必要がある。

## 15. ロール設計

本システムでは、システム全体ロールとプロジェクト内ロールを分けて扱う。

### システム全体ロール

| ロール | 内容 |
|---|---|
| SYSTEM_ADMIN | システム全体を管理する |
| USER | 通常ユーザー |

### プロジェクト内ロール

| ロール | 内容 |
|---|---|
| PROJECT_ADMIN | 対象プロジェクトを管理する |
| MEMBER | 対象プロジェクト内でタスク操作を行う |
| VIEWER | 対象プロジェクト内の情報を閲覧する |

システム管理者とプロジェクト管理者は兼任できる。

システム管理者であるかどうかは、ユーザーに紐づくシステムロールで判定する。

プロジェクト管理者であるかどうかは、対象プロジェクトのプロジェクトメンバー情報で判定する。

プロジェクト作成直後は、作成したシステム管理者を対象プロジェクトのPROJECT_ADMINとして登録する。

システム管理者は、作成後に任意のユーザーをPROJECT_ADMINへ変更できる。

## 16. 認可判定方針

APIでは、以下の順序で認可を判定する。

1. ログイン済みか確認する
2. 対象リソースが存在するか確認する
3. 対象プロジェクトに参加しているか確認する
4. 必要なシステムロールまたはプロジェクト内ロールを持つか確認する
5. コメント投稿者本人など、本人条件が必要な場合は確認する
6. 業務制約を満たすか確認する

ただし、APIの種類によって不要な判定は省略する。

プロジェクト作成では、作成対象のプロジェクトがまだ存在しないため、対象プロジェクト参加確認は行わない。

プロジェクト作成時は、ログインユーザーがシステム管理者であることを確認する。

システム管理者による全プロジェクト閲覧では、対象プロジェクト参加確認は行わない。

未認証で利用するAPIでは、ログイン済み確認は行わない。

認可に失敗した場合は403を返す。

未ログインの場合は401を返す。

対象データが存在しない場合は404を返す。

管理者が0人になる操作など、業務制約に違反する場合は409を返す。

## 18. 管理者不在防止

システム全体でシステム管理者が0人になる操作は禁止する。

各プロジェクト内でプロジェクト管理者が0人になる操作は禁止する。

システム管理者自身のロール変更は、他にシステム管理者が存在する場合のみ可能とする。

プロジェクト管理者自身のロール変更は、対象プロジェクト内に他のプロジェクト管理者が存在する場合のみ可能とする。

プロジェクト管理者の削除は、削除後も対象プロジェクト内にプロジェクト管理者が1人以上残る場合のみ可能とする。

この制約に違反する場合、APIは409を返す。

## 19. 削除済みユーザーの扱い

ユーザー削除は論理削除とする。

ユーザーが削除されても、作成済みのタスク、コメント、添付ファイル、プロジェクトは削除しない。

削除済みユーザーを画面に表示する場合、表示名は「削除済ユーザー」とする。

削除済みユーザーはログインできない。

## 20. 画面側の認可制御

フロントエンドでは、ログインユーザー情報と画面ごとのプロジェクト内ロールに応じて、ボタンやリンクの表示を制御する。

権限のない操作は、原則として非表示または無効化する。

ただし、画面側の制御は補助であり、バックエンドAPI側で必ず認可チェックを行う。

画面側だけで権限を保証してはならない。

## 21. APIエラー時の画面制御

| HTTPステータス | 画面制御 |
|---|---|
| 401 | ログイン画面へ誘導する |
| 403 | 権限がない旨を表示する |
| 404 | 対象データが存在しない旨を表示する |
| 409 | 業務制約により操作できない旨を表示する |
| 500 | サーバーエラーが発生した旨を表示する |

## 22. セキュリティ上の注意点

ログイン失敗時は、メールアドレスが存在するかどうかを推測できる表示にしない。

パスワードリセット申請時は、対象メールアドレスの存在有無に関わらず同じ応答にする。

Cookieにはユーザー情報やロール情報を保存しない。

パスワード、リセットトークン、招待トークンを画面やログに不用意に表示しない。

状態変更APIではCSRF対策を行う。

権限チェックはフロントエンドだけでなく、バックエンドで必ず行う。
