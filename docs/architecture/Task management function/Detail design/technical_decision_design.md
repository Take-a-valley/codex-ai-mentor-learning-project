# 技術詳細・実装方針設計書

## 1. 本書の目的

本書は、タスク管理機能の詳細設計および実装に入る前に、使用技術、実装方針、開発上の前提を整理することを目的とする。

基本設計書で定義した内容をもとに、フロントエンド、バックエンド、DB、認証・認可、メール送信、テストに関する技術的な判断を明確にする。

本書で決定した内容は、以降に作成する詳細設計書および実装時の前提とする。

## 2. 対象範囲

本書では、初期MVPであるカンバン方式タスク管理機能に関する技術詳細を対象とする。

対象は以下とする。

- フロントエンド技術
- バックエンド技術
- DB技術
- 認証・認可方式
- セッション管理
- CSRF対策
- メール送信
- 添付ファイル保存
- テスト方針
- 開発環境
- 今後の詳細設計で前提とする事項

以下は本書の対象外とする。

- Wiki機能
- WBS、ガントチャート機能
- プロジェクト内チャット機能
- 本番インフラ設計
- CI/CD設計
- 監視設計
- 運用設計

## 3. 技術選定の基本方針

技術選定では、以下を重視する。

- 学習しながら理解できること
- 初期MVPを過度に複雑にしないこと
- 一般的な業務アプリケーション開発に近い構成であること
- 認証・認可・DB操作・API設計を学びやすいこと
- 将来、Wiki、WBS、チャット機能を追加しやすいこと
- フロントエンドとバックエンドの責務が分かりやすいこと
- 初期段階ではクラウド依存を増やしすぎないこと

## 5. フロントエンド技術方針

フロントエンドはNext.jsを使用する。

言語はTypeScriptを使用する。

フロントエンドは以下を担当する。

- 画面表示
- ユーザー入力の受付
- 入力値の基本チェック
- API呼び出し
- APIレスポンスの画面反映
- 画面遷移
- ローディング表示
- エラー表示
- ロールに応じた表示制御
- モーダル、確認ダイアログの表示制御

ただし、認証・認可・業務ルールの最終判定はバックエンドで行う。

## 6. Next.jsの利用方針

Next.jsでは、画面単位でルーティングを管理する。

初期MVPでは、以下の画面を実装対象とする。

- ログイン画面
- パスワードリセット申請画面
- パスワード再設定画面
- プロジェクト一覧画面
- プロジェクト作成画面
- プロジェクト編集画面
- カンバン画面
- タスク詳細画面
- メンバー管理画面
- メンバー招待画面
- 招待参加画面

ルーティング方式、ディレクトリ構成、状態管理方式は、フロントエンド詳細設計書で具体化する。

## 7. フロントエンド状態管理方針

フロントエンドでは、以下の状態を扱う。

- ログインユーザー情報
- プロジェクト一覧
- 選択中プロジェクト
- カンバン表示用タスク一覧
- タスク詳細
- コメント一覧
- 添付ファイル一覧
- メンバー一覧
- モーダル表示状態
- ローディング状態
- エラー状態
- CSRFトークン

サーバー状態はTanStack Queryで管理し、画面状態はReact標準の状態管理で扱う。

初期MVPでは、必要以上に複雑な状態管理を避ける。

## 8. UIライブラリ方針

UIはTailwind CSSを基本として採用する。

フォーム、モーダル、ダイアログ、テーブルなど、実装効率や一貫性が必要な箇所では、必要に応じてshadcn/uiを採用する。

初期MVPでは、過度にデザイン依存せず、業務画面として理解しやすいUIを優先する。

UIライブラリを採用する場合でも、業務ロジックはUIコンポーネントに持たせない。

## 9. バックエンド技術方針

バックエンドはSpring Bootを使用する。

言語はJavaを使用する。

バックエンドは以下を担当する。

- APIリクエストの受付
- 認証状態の確認
- 認可チェック
- 入力値の検証
- 業務ルールの実行
- DB操作
- トランザクション制御
- セッション管理
- CSRF対策
- メール送信処理
- 添付ファイル保存処理
- エラーレスポンス返却

## 10. Java / Spring Bootバージョン方針

Javaは、現在PCにインストールされている Java 25 を採用する。

Spring Bootは、Java 25に対応している Spring Boot 4.1.x 系を採用する。

ビルドツールは、現在PCにインストールされている Maven 3.9.9 を採用する。

| 項目 | 採用内容 |
|---|---|
| Java | 25 |
| Spring Boot | 4.1.x |
| ビルドツール | Maven 3.9.9 |

Gradleは初期MVPでは採用しない。

## 11. バックエンド層構成方針

バックエンドは以下の層に分ける。

| 層 | 役割 |
|---|---|
| Controller | HTTPリクエストを受け取り、Serviceを呼び出す |
| Service | 業務処理、認可チェック、トランザクション制御を行う |
| Repository | DBアクセスを行う |
| Entity | DBテーブルに対応するデータ構造を表す |
| DTO | APIリクエスト、レスポンス用のデータ構造を表す |
| Security | 認証、認可、CSRF、セッション制御を扱う |
| Exception | 業務エラー、入力エラー、システムエラーを扱う |

Controllerには業務判断を持たせない。

業務ルールはServiceに集約する。

RepositoryはDBアクセスに責務を限定する。

## 12. DBアクセス方式方針

DBアクセス方式は Spring Data JPA を採用する。

採用理由は以下とする。

- Spring Bootとの相性がよい
- Entity設計とDB設計を対応させやすい
- Repository層を少ない実装量で構成しやすい
- 初期MVPのCRUD実装を進めやすい
- Spring Bootの標準的な構成を学びやすい

論理削除については、Repository設計で `deleted_at is null` を通常取得条件に含める方針とする。

複雑な検索条件が必要になった場合は、Spring Data JPAのクエリメソッド、JPQL、またはカスタムRepositoryで対応する。

## 13. DB方針

DBはMySQLを使用する。

DBでは以下のデータを管理する。

- ユーザー
- システムロール
- プロジェクト
- プロジェクトメンバー
- プロジェクトロール
- タスク
- タスクStatus
- タスクPriority
- コメント
- 添付ファイル
- 招待情報
- パスワードリセット情報

削除は物理削除ではなく論理削除を基本とする。

通常の一覧取得や詳細取得では、論理削除済みデータを返却しない。

## 14. 添付ファイル保存方針

添付ファイル本体はDBに保存する。

添付ファイルでは以下を保存する。

- ファイル名
- MIMEタイプ
- ファイルサイズ
- ファイル本体データ
- アップロード者
- アップロード日時
- 削除日時
- 削除者

1ファイルあたりの容量上限は5MBとする。

許可するファイル種別は以下とする。

- 画像
- PDF
- テキスト
- Office系ファイル

ファイルサイズとファイル種別は、フロントエンドとバックエンドの両方で確認する。

最終的な判定はバックエンドで行う。

## 15. API方式方針

API方式はREST APIとする。

データ形式はJSONを基本とする。

添付ファイル取得APIのみ、ファイルデータを返却する。

状態変更APIでは、CSRFトークンを送信する。

対象となる状態変更APIは以下とする。

- POST
- PUT
- PATCH
- DELETE

APIの詳細なパス、リクエスト、レスポンスはAPI設計書に従う。

## 16. 認証方式方針

認証方式はセッション方式とする。

ログイン成功時、バックエンドはサーバー側でセッションを作成する。

ブラウザはCookieを保持し、以降の認証必須APIではCookieを送信する。

バックエンドはCookieに紐づくセッションからログインユーザーを判定する。

未ログイン時に認証必須APIへアクセスした場合は、401 Unauthorizedを返却する。

ログアウト時はセッションを無効化する。

## 17. Cookie方針

セッションCookie名は `JSESSIONID` を基本とする。

現時点では以下を方針とする。

- セッションIDはJSONレスポンスでは返さない
- セッションIDはCookieで管理する
- Cookieには安全な属性を設定する
- Cookie名は `JSESSIONID` を基本とする
- Cookieの有効期限はセッション有効期限に従う

セッション有効期限は、最終アクセスから120分とする。

## 18. CSRF対策方針

セッション方式を採用するため、CSRF対策を行う。

CSRFトークンの生成と検証はSpring Securityで行う。

フロントエンドがCSRFトークンを取得できるように、CSRFトークン取得APIを用意する。

状態変更APIではCSRFトークンを必須とする。

対象となるHTTPメソッドは以下とする。

- POST
- PUT
- PATCH
- DELETE

CSRFトークンが不足している場合、または不正な場合は403 Forbiddenを返却する。

CSRFトークン取得APIのパスは `/api/csrf-token` とする。

レスポンス形式は `CsrfTokenResponse` とし、CSRFトークンのヘッダー名はレスポンスの `headerName` を使用する。

## 19. 認可方針

認可はバックエンドで必ず実施する。

フロントエンドの表示制御は、ユーザー体験向上のための補助とする。

認可判定では以下を使用する。

- ログイン状態
- システムロール
- 対象プロジェクトへの参加状態
- 対象プロジェクト内ロール
- 操作対象データの所有者
- 論理削除状態
- 管理者不在防止などの業務制約

システム管理者は、全プロジェクトに対する管理操作を行える。

プロジェクト管理者は、対象プロジェクト内の管理操作を行える。

メンバーは、対象プロジェクト内のタスク操作、コメント操作、添付ファイル追加を行える。

閲覧者は、対象プロジェクト内の情報を閲覧できる。

## 20. メール送信方針

初期MVPでは、以下のメール送信を扱う。

- パスワードリセットメール
- プロジェクト招待メール

開発環境では、実メール送信ではなくMailpitを使用する。

MailpitはDocker Composeで起動し、送信されたメールをブラウザ上で確認できるようにする。

パスワードリセットメールには、30分間有効なリセットURLを含める。

プロジェクト招待メールには、24時間有効な招待URLを含める。

本番向けのメール送信方式は、初期MVPの実装後または運用設計時に具体化する。

## 21. トークン管理方針

パスワードリセットとプロジェクト招待では、推測困難なトークンを発行する。

トークンでは以下を管理する。

- トークン値
- 対象ユーザーまたはメールアドレス
- 対象プロジェクト
- 有効期限
- 使用済み状態
- 作成日時
- 更新日時

使用済みトークンは再利用できない。

期限切れトークンは利用できない。

期限切れ状態を自動更新するか、参照時に判定するかは後続工程で決定する。

## 22. 論理削除方針

主要データの削除は論理削除とする。

論理削除では、削除日時または削除状態を更新する。

通常の一覧取得や詳細取得では、論理削除済みデータを返却しない。

ユーザー削除時は、作成済みタスクやコメントは削除しない。

削除済みユーザーの表示名は「削除済ユーザー」とする。

プロジェクト削除時は、対象プロジェクトに紐づくタスク、コメント、添付ファイルも論理削除扱いとする。

招待情報とパスワードリセット情報は履歴として残す。

## 23. エラー制御方針

APIでは、エラー内容に応じて適切なHTTPステータスを返却する。

| HTTPステータス | 用途 |
|---|---|
| 400 | 入力値不正 |
| 401 | 未認証 |
| 403 | 権限不足、CSRF不正 |
| 404 | 対象データなし |
| 409 | 業務制約違反 |
| 413 | ファイルサイズ超過 |
| 415 | 許可されていないファイル種別 |
| 500 | サーバー内部エラー |

エラーレスポンスの形式はAPI設計書に従う。

## 24. ログ方針

バックエンドでは、運用上必要なログを出力する。

対象例は以下とする。

- ログイン成功、失敗
- ログアウト
- プロジェクト作成、更新、削除
- ロール変更
- 招待作成、承認
- パスワードリセット申請、完了
- 業務エラー
- システムエラー

ログには以下を出力しない。

- パスワード
- セッションID
- CSRFトークン
- パスワードリセットトークン
- 招待トークン
- 添付ファイル本体

## 25. テスト技術方針

初期MVPでは、以下のテスト技術を採用候補とする。

| 対象 | 採用候補 |
|---|---|
| バックエンド単体テスト | Spring Boot Test / JUnit |
| APIテスト | Spring Boot Test |
| フロントエンド単体テスト | Vitest + React Testing Library |
| E2Eテスト | Playwright |
| 結合テスト | Spring Boot Test、必要に応じてTestcontainers |
| 回帰テスト | 主要シナリオを中心に実施 |

テストでは、認証、認可、論理削除、PROJECT_ADMINが0人になる操作の禁止、添付ファイル制約を重点的に確認する。

開発環境ではDocker Composeを採用する。

Docker Composeでは、初期MVPで以下を起動する。

- MySQL 8系

|---|---|
| フロントエンド | Node.js 24.14.1 / npm 11.11.0でローカル起動 |

| 添付ファイル | DBに保存する |

将来的に必要になった場合、フロントエンドとバックエンドもDocker化を検討する。
