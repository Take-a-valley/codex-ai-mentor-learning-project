# フロントエンド構成設計書

## 1. 本書の目的

本書は、タスク管理機能の初期MVPにおけるフロントエンドの構成方針を定義することを目的とする。

Next.js、TypeScript、Tailwind CSSを前提に、画面ルーティング、ディレクトリ構成、コンポーネント分割、API呼び出し、認証状態の扱い、エラー表示方針を整理する。

## 2. 対象範囲

対象に含めるものは以下とする。

- Next.jsの利用方針
- 画面ルーティング方針
- ディレクトリ構成方針
- コンポーネント分割方針
- APIクライアント方針
- 認証状態の扱い
- CSRFトークンの扱い
- 権限に応じた表示制御
- エラー表示方針
- ローディング表示方針

対象外とするものは以下とする。

- 詳細な画面レイアウト
- CSSの詳細実装
- 個別コンポーネントの実装コード
- E2Eテスト設計
- 本番デプロイ設計

## 3. 前提技術

| 分類 | 採用技術 |
|---|---|
| フレームワーク | Next.js 16.x または作成時点のlatest |
| 言語 | TypeScript |
| 実行環境 | Node.js 24.14.1 |
| パッケージ管理 | npm 11.11.0 |
| UI | Tailwind CSS |
| UI部品 | 必要に応じてshadcn/ui |
| 状態管理 | React標準、必要に応じてTanStack Query |
| API通信 | fetchを基本とする |
| テスト | Vitest + React Testing Library |

## 4. フロントエンドの責務

フロントエンドは以下を担当する。

- 画面表示
- ユーザー入力の受付
- 入力値の基本チェック
- API呼び出し
- APIレスポンスの画面反映
- ローディング表示
- エラー表示
- モーダル表示制御
- ロールに応じたボタンやメニューの表示制御
- CSRFトークンの取得と送信

認証・認可・業務ルールの最終判定はバックエンドで行う。

## 5. 画面ルーティング方針

初期MVPでは以下の画面を用意する。

| 画面 | 用途 |
|---|---|
| ログイン画面 | メールアドレスとパスワードでログインする |
| パスワードリセット申請画面 | リセットメール送信を申請する |
| パスワード再設定画面 | リセットURLから新しいパスワードを設定する |
| プロジェクト一覧画面 | 参加または管理可能なプロジェクトを一覧表示する |
| プロジェクト作成画面 | SYSTEM_ADMINがプロジェクトを作成する |
| プロジェクト編集画面 | プロジェクト情報を編集する |
| カンバン画面 | タスクをStatus別に表示・移動する |
| タスク詳細画面 | タスク詳細、コメント、添付ファイルを表示する |
| メンバー管理画面 | プロジェクトメンバーとロールを管理する |
| メンバー招待画面 | メールアドレスでメンバーを招待する |
| 招待参加画面 | 招待URLからプロジェクト参加を完了する |

## 6. ルーティング設計方針

画面URLは、機能単位で分かりやすい構成とする。

| 画面 | URL方針 |
|---|---|
| ログイン | `/login` |
| パスワードリセット申請 | `/password-reset/request` |
| パスワード再設定 | `/password-reset/confirm` |
| プロジェクト一覧 | `/projects` |
| プロジェクト作成 | `/projects/create` |
| プロジェクト編集 | `/projects/{projectId}/edit` |
| カンバン | `/projects/{projectId}/board` |
| タスク詳細 | `/projects/{projectId}/tasks/{taskId}` |
| メンバー管理 | `/projects/{projectId}/members` |
| メンバー招待 | `/projects/{projectId}/invitations/new` |
| 招待参加 | `/invitations/{token}` |

## 7. ディレクトリ構成方針

フロントエンドは、画面、共通部品、API通信、型定義、ユーティリティを分けて管理する。

| 分類 | 用途 |
|---|---|
| app | 画面ルーティング |
| components | 共通UIコンポーネント |
| features | 機能単位のコンポーネント、hooks、処理 |
| lib | APIクライアント、共通関数 |
| types | APIレスポンスや画面で使用する型 |
| constants | 固定値 |
| styles | グローバルスタイル |

## 8. コンポーネント分割方針

コンポーネントは責務ごとに分割する。

| 種別 | 役割 |
|---|---|
| Page | ルーティング単位の画面 |
| Feature Component | 業務機能に紐づく表示部品 |
| UI Component | 汎用的なボタン、入力欄、モーダルなど |
| Layout Component | ヘッダー、サイドバー、画面枠 |
| Form Component | 入力フォーム |
| Dialog Component | 確認ダイアログ、作成・編集モーダル |

タスク作成・編集は画面遷移ではなくモーダルで行う。

## 9. API通信方針

API通信は共通APIクライアントを経由して行う。

APIクライアントでは以下を扱う。

- ベースURL
- Cookie送信
- CSRFトークン送信
- JSONリクエスト
- multipart/form-dataリクエスト
- エラーレスポンス処理
- 認証エラー時の扱い

各画面やコンポーネントから直接fetch処理を散在させない。

## 10. 認証状態の扱い

ログイン状態は、バックエンドのセッションを正とする。

フロントエンドでは、ログインユーザー情報取得APIの結果をもとに、ログイン済みかどうかを判定する。

未ログイン状態で認証必須画面へアクセスした場合は、ログイン画面へ誘導する。

ログアウト時はバックエンドのログアウトAPIを呼び出し、画面側のユーザー情報を破棄する。

## 11. CSRFトークン方針

状態変更APIを呼び出す前に、CSRFトークン取得APIからトークンを取得する。

対象となるHTTPメソッドは以下とする。

- POST
- PUT
- PATCH
- DELETE

取得したCSRFトークンは、API設計書で定義されたヘッダー名で送信する。

## 12. 権限による表示制御

フロントエンドでは、ログインユーザーのシステムロールおよびプロジェクトロールに応じて、操作ボタンやメニューの表示を制御する。

| 権限 | 表示制御例 |
|---|---|
| SYSTEM_ADMIN | プロジェクト作成ボタンを表示する |
| PROJECT_ADMIN | メンバー管理、ロール変更、タスク削除を表示する |
| MEMBER | タスク作成・編集・移動、コメント投稿を表示する |
| VIEWER | 閲覧系の操作のみ表示する |

ただし、認可の最終判定は必ずバックエンドで行う。

## 13. カンバン画面構成方針

カンバン画面では、Statusごとにタスク列を表示する。

初期MVPのStatusは以下とする。

- ToDo
- In Progress
- Review
- Done

ドラッグアンドドロップによるタスク移動を初期MVPに含める。

タスク移動時は、移動先Statusをバックエンドへ送信する。

初期MVPでは、ドラッグアンドドロップによる移動はStatus変更を対象とし、同一Status内の並び替えは対象外とする。

## 14. タスク作成・編集モーダル方針

タスク作成・編集はモーダルで行う。

モーダルでは以下を入力・編集できる。

- Summary
- Description
- Status
- Priority
- Assignee
- Due Date

作成・編集完了後は、カンバン画面またはタスク詳細画面の表示内容を更新する。

## 15. コメント・添付ファイル表示方針

タスク詳細画面では、コメントと添付ファイルを表示する。

コメントは作成日時順で表示する。

添付ファイルはファイル名、サイズ、アップロード者、アップロード日時を表示する。

削除済みユーザーが作成したコメントや添付ファイルは、ユーザー名を「削除済ユーザー」として表示する。

コメント操作の表示制御は以下とする。

| 操作 | 表示方針 |
|---|---|
| コメント作成 | システム管理者、プロジェクト管理者、メンバーに表示する |
| コメント編集 | メンバーは自分のコメントのみ表示する。管理者は必要に応じて管理操作を表示する |
| コメント削除 | システム管理者、プロジェクト管理者に表示する |

添付ファイル操作の表示制御は以下とする。

| 操作 | 表示方針 |
|---|---|
| 添付追加 | システム管理者、プロジェクト管理者、メンバーに表示する |
| 添付削除 | システム管理者、プロジェクト管理者に表示する |

## 16. ローディング表示方針

API通信中は、画面または部品単位でローディング状態を表示する。

初期表示、一覧更新、送信処理では、それぞれ適切なローディング表示を行う。

二重送信を防ぐため、送信中のボタンは無効化する。

## 17. エラー表示方針

APIエラーは、画面上でユーザーが理解できる形で表示する。

| エラー | 表示方針 |
|---|---|
| 400 | 入力内容の見直しを促す |
| 401 | ログイン画面へ誘導する |
| 403 | 権限がない旨を表示する |
| 404 | 対象データが存在しない旨を表示する |
| 409 | 業務制約違反の内容を表示する |
| 413 | ファイルサイズ超過を表示する |
| 415 | 許可されていないファイル種別を表示する |
| 500 | サーバーエラーとして表示する |

## 18. 入力チェック方針

フロントエンドでは、入力漏れや明らかな形式不正を事前にチェックする。

ただし、最終的な入力チェックはバックエンドで行う。

フロントエンドの入力チェックは、ユーザー体験を向上させるための補助とする。

## 19. 添付ファイル方針

添付ファイルアップロードでは、multipart/form-dataを使用する。

フロントエンドでは以下を確認する。

- ファイルサイズが5MB以下であること
- 許可されたファイル種別であること
- ファイルが選択されていること

最終判定はバックエンドで行う。

## 20. 決定事項

- フロントエンドはNext.jsとTypeScriptで構成する
- UIはTailwind CSSを基本とする
- 必要に応じてshadcn/uiを使用する
- API通信は共通APIクライアントに集約する
- 認証状態はバックエンドのセッションを正とする
- 状態変更APIではCSRFトークンを送信する
- タスク作成・編集はモーダルで行う
- ドラッグアンドドロップを初期MVPに含める
- 認可の最終判定はバックエンドで行う
