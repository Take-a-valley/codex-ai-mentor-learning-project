# フロントエンドAPIクライアント設計書

## 1. 本書の目的

本書は、タスク管理機能のフロントエンドからバックエンドAPIを呼び出すためのAPIクライアント設計を定義する。

API呼び出し処理を画面やコンポーネントへ直接分散させず、共通のAPIクライアントに集約することで、認証、CSRF、エラー処理、レスポンス処理を一貫して扱えるようにする。

## 2. 対象範囲

対象に含めるものは以下とする。

- APIクライアントの責務
- API呼び出しの共通方針
- Cookie送信方針
- CSRFトークン送信方針
- JSONリクエスト方針
- multipart/form-dataリクエスト方針
- ファイルダウンロード方針
- APIエラー処理方針
- TanStack Queryとの連携方針
- 機能別APIモジュール方針

対象外とするものは以下とする。

- APIサーバー側の実装
- Controller、Service、Repositoryの実装
- 個別画面コンポーネントの実装
- 詳細なテストケース

## 3. 前提

フロントエンドはNext.jsとTypeScriptで構成する。

認証方式はセッション方式とする。

バックエンドはログイン成功時にセッションを作成し、ブラウザはCookieでセッションIDを保持する。

状態変更APIではCSRFトークンを送信する。

CSRFトークンは `/api/csrf-token` から取得し、レスポンスの `headerName` に従って送信する。

## 4. APIクライアントの責務

APIクライアントは以下を担当する。

- APIのベースURL管理
- リクエスト送信
- Cookie送信
- CSRFトークン付与
- JSONリクエスト作成
- multipart/form-dataリクエスト作成
- レスポンスJSONの取り出し
- ファイルレスポンスの取り扱い
- HTTPステータスに応じたエラー変換
- 共通エラー形式への変換
- 認証エラー、認可エラーの判定

APIクライアントは、業務判断や画面表示判断を持たない。

## 5. 配置方針

APIクライアントは、フロントエンド共通処理として `lib` または `features` 配下に配置する。

配置方針は以下とする。

| 分類 | 役割 |
|---|---|
| 共通APIクライアント | fetch、Cookie、CSRF、エラー処理を扱う |
| 機能別APIモジュール | auth、projects、tasksなど機能単位のAPI呼び出しを定義する |
| API型定義 | リクエストDTO、レスポンスDTOに対応する型を定義する |
| Query Hooks | TanStack Queryから機能別APIを利用する |

## 6. API呼び出しの基本方針

API呼び出しは共通APIクライアントを経由する。

画面コンポーネントから直接fetchを呼び出さない。

画面コンポーネントは、Query Hooksまたは機能別APIモジュールを通じてAPIを利用する。

## 7. ベースURL方針

開発環境では、フロントエンドとバックエンドが別ポートで起動する想定とする。

APIのベースURLは環境変数で管理する。

環境ごとにAPI接続先を切り替えられるようにする。

## 8. Cookie送信方針

APIクライアントでは、Cookieを送信できる設定でAPIを呼び出す。

開発環境ではフロントエンドとバックエンドが別ポートで起動する想定のため、Cookie送信設定を明示する。

認証方式はセッション方式のため、認証必須APIではブラウザが保持している `JSESSIONID` をバックエンドへ送信する。

セッションCookieはHttpOnlyで管理するため、フロントエンドのJavaScriptからセッションIDを読み取らない。

## 9. CSRFトークン方針

状態変更APIではCSRFトークンを送信する。

対象となるHTTPメソッドは以下とする。

- POST
- PUT
- PATCH
- DELETE

CSRFトークンは `/api/csrf-token` から取得する。

APIクライアントは、取得したCSRFトークンとヘッダー名を保持し、状態変更APIに付与する。

CSRFトークンが未取得の場合、状態変更APIの前に取得する。

CSRFトークンが無効になった場合は、再取得してから再実行を検討する。

ログイン、パスワードリセット申請、パスワード再設定、招待参加など、認証不要のPOST APIでもCSRFトークンを送信する。

CSRF要否は認証要否とは別に判断する。

## 10. JSONリクエスト方針

通常のAPIリクエストはJSON形式で送信する。

対象例は以下とする。

- ログイン
- パスワードリセット申請
- パスワード再設定
- プロジェクト作成
- プロジェクト編集
- タスク作成
- タスク編集
- Status変更
- コメント作成
- コメント編集
- ロール変更
- 招待作成

JSONリクエストでは、API設計書で定義したリクエスト項目を送信する。

## 11. multipart/form-data方針

添付ファイルアップロードでは `multipart/form-data` を使用する。

フロントエンドでは、送信前に以下を確認する。

- ファイルが選択されていること
- 1ファイルあたり5MB以下であること
- 許可されたファイル種別であること

ただし、最終判定はバックエンドで行う。

## 12. ファイルダウンロード方針

添付ファイル取得APIでは、ファイル本体をレスポンスとして受け取る。

APIクライアントは、JSONではなくファイルデータとして扱う。

画面側では、ファイル名、MIMEタイプ、サイズなどをもとにダウンロードまたは表示を行う。

## 13. レスポンス処理方針

正常レスポンスでは、API設計書で定義したJSON構造を受け取る。

APIクライアントはレスポンスを取り出し、呼び出し元へ返す。

画面表示用の加工は、必要に応じて呼び出し側または画面用の変換処理で行う。

APIクライアント内に画面固有の表示文言を持たせない。

## 14. エラー処理方針

APIエラーは、共通エラー形式として扱う。

HTTPステータスごとの基本方針は以下とする。

| HTTPステータス | フロントエンドでの扱い |
|---|---|
| 400 | 入力エラーとして画面に表示する |
| 401 | 未ログインとしてログイン画面へ誘導する |
| 403 | 権限不足またはCSRF不正として表示する |
| 404 | 対象データなしとして表示する |
| 409 | 業務制約違反として表示する |
| 413 | ファイルサイズ超過として表示する |
| 415 | 許可されていないファイル種別として表示する |
| 500 | サーバーエラーとして表示する |

## 15. 認証エラー処理方針

401が返却された場合、ログイン状態が無効であると扱う。

認証必須画面では、ログイン画面へ誘導する。

ログイン画面、パスワードリセット画面、招待参加画面では、認証必須画面と同じレイアウトを表示しない。

## 16. 認可エラー処理方針

403が返却された場合は、エラーレスポンスのerrorCodeを確認する。

`ACCESS_DENIED` の場合は権限不足として扱う。

`CSRF_INVALID` の場合はCSRFトークン不正として扱い、必要に応じてCSRFトークンを再取得する。

権限不足の場合は、操作できない旨を表示する。

CSRF不正の場合は、CSRFトークンの再取得を検討する。

最終的な判定はバックエンドのレスポンス内容に従う。

## 17. TanStack Query連携方針

サーバー状態はTanStack Queryで管理する。

対象となるサーバー状態は以下とする。

- ログインユーザー情報
- プロジェクト一覧
- プロジェクト詳細
- タスク一覧
- タスク詳細
- コメント一覧
- 添付ファイル一覧
- メンバー一覧
- 招待情報

取得系APIはQueryとして扱う。

更新系APIはMutationとして扱う。

Mutation成功後は、関連するQueryを再取得する。

## 18. Query Key方針

Query Keyは、機能とIDが分かる形で設計する。

対象データの単位ごとにQuery Keyを分ける。

例として、以下の単位で管理する。

- ログインユーザー
- プロジェクト一覧
- プロジェクト詳細
- プロジェクト内タスク一覧
- タスク詳細
- タスクコメント一覧
- タスク添付ファイル一覧
- プロジェクトメンバー一覧

Query Keyは、配列形式で定義する。

先頭要素に機能名、後続要素に識別子や検索条件を含める。

例として、プロジェクト詳細は project と projectId、タスク一覧は tasks と projectId と検索条件を組み合わせる。

## 19. Mutation後の再取得方針

データ更新後は、画面に古い情報が残らないように関連データを再取得する。

代表例は以下とする。

| 操作 | 再取得対象 |
|---|---|
| プロジェクト作成 | プロジェクト一覧 |
| プロジェクト編集 | プロジェクト詳細、プロジェクト一覧 |
| タスク作成 | タスク一覧 |
| タスク編集 | タスク一覧、タスク詳細 |
| Status変更 | タスク一覧、タスク詳細 |
| コメント作成 | コメント一覧、タスク詳細 |
| 添付追加 | 添付ファイル一覧、タスク詳細 |
| メンバー変更 | メンバー一覧、ログインユーザーの権限情報 |

## 20. 機能別APIモジュール方針

API呼び出しは機能別に分ける。

| モジュール | 対象 |
|---|---|
| authApi | ログイン、ログアウト、ログインユーザー取得 |
| csrfApi | CSRFトークン取得 |
| passwordResetApi | パスワードリセット申請、再設定 |
| projectApi | プロジェクト一覧、作成、詳細、編集、削除 |
| taskApi | タスク一覧、作成、詳細、編集、Status変更、削除 |
| commentApi | コメント作成、編集、削除 |
| attachmentApi | 添付ファイル追加、取得、削除 |
| memberApi | メンバー一覧、ロール変更、削除 |
| invitationApi | 招待作成、招待確認、招待参加 |

## 21. ログ出力方針

フロントエンドでは、開発時の確認に必要な範囲でログを出力する。

以下はログへ出力しない。

- パスワード
- セッションID
- CSRFトークン
- 招待トークン
- パスワードリセットトークン
- 添付ファイル本体

## 22. テスト観点

APIクライアントでは以下を確認する。

- GETリクエストが正しく送信されること
- POST、PUT、PATCH、DELETEでCSRFトークンが付与されること
- Cookie送信設定が有効であること
- JSONリクエストを送信できること
- multipart/form-dataを送信できること
- ファイルレスポンスを扱えること
- 400、401、403、404、409、413、415、500を適切に扱えること
- Mutation成功後に関連Queryを再取得できること

## 23. 決定事項

- API呼び出しは共通APIクライアントに集約する
- 画面コンポーネントから直接fetchを呼び出さない
- 認証はセッションCookieを使用する
- セッションIDはJavaScript上で直接扱わない
- 状態変更APIではCSRFトークンを送信する
- CSRFトークンのヘッダー名はCsrfTokenResponseのheaderNameに従う
- サーバー状態はTanStack Queryで管理する
- 更新処理後は関連Queryを再取得する
- 添付ファイルアップロードはmultipart/form-dataを使用する
- 添付ファイル取得はファイルレスポンスとして扱う
