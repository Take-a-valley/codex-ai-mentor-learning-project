# セキュリティ設計書

## 1. 本書の目的

本書は、タスク管理機能の初期MVPにおけるセキュリティ詳細設計を定義することを目的とする。

本書では、認証、認可、セッション管理、Cookie、CSRF対策、パスワード管理、トークン管理、ログ出力、API保護方針を整理する。

## 2. 対象範囲

対象に含めるものは以下とする。

- 認証方式
- セッション管理
- Cookie管理
- CSRF対策
- 認可判定
- システムロール
- プロジェクトロール
- パスワード管理
- パスワードリセット
- 招待トークン
- 削除済みユーザーの扱い
- API別の認証要否
- セキュリティ関連エラー
- ログ出力方針

対象外とするものは以下とする。

- OAuthログイン
- SNSログイン
- SSO
- 多要素認証
- JWT認証
- APIキー認証
- 本番メール基盤の詳細
- インフラレベルのWAF設定
- 脆弱性診断の詳細手順

## 3. セキュリティ設計方針

認証方式はセッション方式を採用する。

認証、CSRF、セッション管理はSpring Securityを中心に扱う。

認可判定はService層で必ず行う。

フロントエンドの表示制御だけで権限を保証しない。

Cookieにはユーザー情報やロール情報を保存しない。

パスワード、セッションID、CSRFトークン、招待トークン、パスワードリセットトークンをログに出力しない。

## 4. 認証方式

本システムでは、メールアドレスとパスワードによるログインを行う。

認証方式はセッション方式とする。

ログイン成功時、バックエンドはサーバー側にセッションを作成する。

ブラウザはセッションCookieを保持し、以降の認証必須APIリクエストでCookieを送信する。

バックエンドはCookieに紐づくセッション情報からログインユーザーを判定する。

## 5. セッション管理

セッションはSpring Securityの仕組みを利用して管理する。

セッションには、ログイン済みユーザーを識別するために必要な情報のみを保持する。

セッション有効期限は、最終アクセスから120分とする。

| 項目 | 方針 |
|---|---|
| 認証方式 | セッション方式 |
| セッション有効期限 | 最終アクセスから120分 |
| ログアウト時 | セッションを無効化する |
| セッション期限切れ時 | 401 Unauthorized |
| 削除済みユーザー | 認証済みユーザーとして扱わない |

## 6. Cookie管理

セッションCookieにはセッション識別子のみを保存する。

ユーザーID、メールアドレス、表示名、ロール情報はCookieに保存しない。

| 項目 | 方針 |
|---|---|
| Cookie名 | JSESSIONIDを基本とする |
| HttpOnly | 有効 |
| Secure | 本番環境では有効 |
| SameSite | Laxを基本とする |
| Path | / |
| 有効期限 | セッション有効期限に従う |

HttpOnlyを有効にし、JavaScriptからセッションCookieを直接参照できないようにする。

SecureはHTTPS通信を前提とする本番環境で有効にする。

SameSiteはLaxを基本とし、外部サイトからの意図しないリクエストを抑制する。

## 7. CSRF対策

セッションCookieを利用するため、CSRF対策を行う。

CSRFトークンの生成と検証はSpring Securityで行う。

フロントエンドは、状態変更APIを呼び出す前にCSRFトークンを取得する。

CSRFトークン取得APIは以下とする。

| 項目 | 内容 |
|---|---|
| メソッド | GET |
| パス | /api/csrf-token |
| 認証 | 不要 |
| CSRF | 不要 |
| レスポンスDTO | CsrfTokenResponse |

CSRFトークンは、状態変更APIのリクエストヘッダーに設定する。

CSRFトークンのヘッダー名は、CsrfTokenResponseのheaderNameで返却する値を使用する。

## 8. CSRF対象API

状態変更を行うAPIではCSRFトークンを必須とする。

| HTTPメソッド | CSRF要否 |
|---|---|
| GET | 不要 |
| POST | 必要 |
| PUT | 必要 |
| PATCH | 必要 |
| DELETE | 必要 |

GET APIでは状態変更を行わない。

CSRFトークンが不足または不正な場合は、403 Forbiddenを返却する。

## 9. 認証不要API

以下のAPIは未ログイン状態でも利用できる。

| API | 用途 |
|---|---|
| POST /api/auth/login | ログイン |
| GET /api/csrf-token | CSRFトークン取得 |
| POST /api/password-reset/request | パスワードリセット申請 |
| POST /api/password-reset/confirm | パスワード再設定 |
| GET /api/invitations/{token} | 招待情報取得 |
| POST /api/invitations/{token}/register | 未登録ユーザー登録・招待承認 |

上記以外のAPIは、原則として認証必須とする。

## 10. 認証必須API

認証必須APIでは、セッションからログインユーザーを取得する。

未ログイン、セッション期限切れ、削除済みユーザーの場合は401 Unauthorizedを返却する。

認証必須APIでは、ControllerまたはSpring Securityの処理を通じてログインユーザー情報をServiceへ渡す。

最終的な認可判定はService層で行う。

## 11. ロール設計

本システムでは、システム全体ロールとプロジェクト内ロールを分ける。

### システムロール

| ロール | 内容 |
|---|---|
| SYSTEM_ADMIN | システム全体を管理する |
| USER | 通常ユーザー |

### プロジェクトロール

| ロール | 内容 |
|---|---|
| PROJECT_ADMIN | 対象プロジェクトを管理する |
| MEMBER | 対象プロジェクト内でタスク操作を行う |
| VIEWER | 対象プロジェクト内の情報を閲覧する |

システム管理者とプロジェクト管理者は兼任できる。

システム管理者かどうかはUserに紐づくシステムロールで判定する。

プロジェクト管理者かどうかはProjectMemberに紐づくプロジェクトロールで判定する。

## 12. 認可判定方針

認可判定はService層で行う。

APIごとに必要なシステムロールまたはプロジェクトロールを確認する。

画面側のボタン表示制御は補助であり、バックエンド側の認可判定を必須とする。

認可判定では以下を確認する。

| 判定 | 内容 |
|---|---|
| ログイン確認 | 認証必須APIでログイン済みか確認する |
| 対象データ確認 | 操作対象データが存在し、論理削除されていないか確認する |
| プロジェクト参加確認 | 対象プロジェクトに参加しているか確認する |
| システムロール確認 | SYSTEM_ADMINか確認する |
| プロジェクトロール確認 | PROJECT_ADMIN、MEMBER、VIEWERを確認する |
| 本人確認 | コメント更新など本人条件がある操作で確認する |
| 業務制約確認 | 管理者0人禁止などを確認する |

| メンバー招待 | SYSTEM_ADMINまたはPROJECT_ADMIN |
| ロール変更 | SYSTEM_ADMINまたはPROJECT_ADMIN |
| メンバー削除 | SYSTEM_ADMINまたはPROJECT_ADMIN |
| タスク一覧取得 | SYSTEM_ADMINまたは対象プロジェクト参加者 |
| タスク詳細取得 | SYSTEM_ADMINまたは対象プロジェクト参加者 |

## 14. 管理者不在防止

システム全体でSYSTEM_ADMINが0人になる操作は禁止する。

各プロジェクト内でPROJECT_ADMINが0人になる操作は禁止する。

PROJECT_ADMINのロール変更またはメンバー削除では、操作後も対象プロジェクトにPROJECT_ADMINが1人以上残ることを確認する。

制約に違反する場合は409 Conflictを返却する。

## 15. プロジェクト作成時の権限処理

プロジェクト作成はSYSTEM_ADMINのみ可能とする。

プロジェクト作成直後、作成したSYSTEM_ADMINを対象プロジェクトのPROJECT_ADMINとしてProjectMemberに登録する。

SYSTEM_ADMINは、作成後に任意のユーザーをPROJECT_ADMINへ変更できる。

プロジェクト内のPROJECT_ADMINが0人になるロール変更は許可しない。

## 16. パスワード管理

パスワードは平文で保存しない。

パスワードはSpring Securityで利用可能な安全な方式でハッシュ化して保存する。

ログイン時は、入力されたパスワードと保存済みハッシュを比較する。

パスワード再設定時は、新しいパスワードをハッシュ化して保存する。

パスワードはAPIレスポンスに含めない。

パスワードはログに出力しない。

## 17. パスワードリセット

パスワードリセットURLの有効期限は30分とする。

パスワードリセットトークンは推測困難な値とする。

パスワードリセット申請では、対象メールアドレスの存在有無に関わらず同じレスポンスを返す。

論理削除済みユーザーはパスワードリセット対象外とする。

パスワードリセットトークンのstatusは以下とする。

| status | 内容 |
|---|---|
| ACTIVE | 使用可能 |
| USED | 使用済み |
| EXPIRED | 期限切れ |

使用済みまたは期限切れのトークンは利用できない。

## 18. 招待トークン

招待URLの有効期限は24時間とする。

招待トークンは推測困難な値とする。

招待時の初期ロールはMEMBER固定とする。

招待トークンのstatusは以下とする。

| status | 内容 |
|---|---|
| PENDING | 招待中 |
| ACCEPTED | 承認済み |
| EXPIRED | 期限切れ |
| REVOKED | 無効化済み |

登録済みユーザーが招待を承認する場合、招待先メールアドレスとログインユーザーのメールアドレスが一致することを確認する。

未登録ユーザー登録時に、招待先emailのユーザーが既に存在する場合は、新規登録せず登録済みユーザー向けの招待承認を利用させる。

## 19. 削除済みユーザーの扱い

ユーザー削除は論理削除とする。

論理削除済みユーザーはログインできない。

論理削除済みユーザーはパスワードリセット対象外とする。

削除済みユーザーが作成したタスク、コメント、添付ファイルは削除しない。

削除済みユーザーを画面に表示する場合、表示名は `削除済ユーザー` とする。

## 20. 論理削除済みデータの扱い

論理削除済みデータは、通常操作では存在しないものとして扱う。

論理削除済みプロジェクト配下のタスク、コメント、添付ファイルは操作対象外とする。

論理削除済みタスクにはコメントや添付ファイルを追加できない。

論理削除済み添付ファイルは取得できない。

## 21. セキュリティ関連エラー

| 発生条件 | HTTPステータス | エラーコード |
|---|---|---|
| 未ログイン | 401 | UNAUTHENTICATED |
| セッション期限切れ | 401 | UNAUTHENTICATED |
| 削除済みユーザーによるアクセス | 401 | UNAUTHENTICATED |
| 権限不足 | 403 | ACCESS_DENIED |
| CSRFトークン不正 | 403 | CSRF_INVALID |
| token不正 | 400 | TOKEN_INVALID |
| token期限切れ | 409 | TOKEN_EXPIRED |
| 管理者0人禁止違反 | 409 | BUSINESS_RULE_VIOLATION |

詳細は例外設計書に従う。

## 22. フロントエンド側の扱い

フロントエンドは、ログイン状態確認のために `GET /api/auth/me` を使用する。

フロントエンドは、状態変更APIの前にCSRFトークンを取得し、リクエストヘッダーに設定する。

フロントエンドは、ログインユーザー情報と対象プロジェクト内ロールに応じて、ボタンやリンクの表示を制御する。

ただし、フロントエンド側の表示制御は補助であり、バックエンド側の認可判定を必ず行う。

## 23. ログ出力方針

以下の情報はログに出力しない。

- パスワード
- セッションID
- CSRFトークン
- パスワードリセットトークン
- 招待トークン

認証失敗、認可失敗、CSRFエラー、token不正などは、調査に必要な範囲でログに記録する。

個人情報をログに出力する場合は、必要最小限にする。

## 24. セキュリティテスト観点

| 観点 | 確認内容 |
|---|---|
| 未ログインアクセス | 認証必須APIで401になること |
| 権限不足 | 権限のない操作で403になること |
| CSRF | 状態変更APIでCSRFトークン不正時に403になること |
| セッション期限切れ | 期限切れ後に401になること |
| ログアウト | ログアウト後に認証必須APIへアクセスできないこと |
| パスワードリセット | 期限切れ、使用済みtokenが利用できないこと |
| 招待 | 期限切れ、承認済み、無効化済みtokenが利用できないこと |
| 管理者0人禁止 | PROJECT_ADMINが0人になる操作が409になること |
| 削除済みユーザー | ログインできないこと |
| 論理削除済みデータ | 通常操作対象外になること |

## 25. 決定事項

- 認証方式はセッション方式とする
- Spring Securityを利用する
- セッション有効期限は最終アクセスから120分とする
- セッションCookie名はJSESSIONIDを基本とする
- CookieはHttpOnlyを有効にする
- CookieのSameSiteはLaxを基本とする
- 本番環境ではCookieのSecureを有効にする
- CSRFトークン取得APIは `/api/csrf-token` とする
- 状態変更APIではCSRFトークンを必須とする
- 認可判定はService層で行う
- パスワードはハッシュ化して保存する
- パスワードリセットURLの有効期限は30分とする
- 招待URLの有効期限は24時間とする
- tokenやセッションIDはログに出力しない
