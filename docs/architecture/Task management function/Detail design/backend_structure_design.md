# バックエンド構成設計書

## 1. 本書の目的

本書は、タスク管理機能の初期MVPにおけるバックエンドの構成方針を定義することを目的とする。

Spring Bootで実装するバックエンドについて、パッケージ構成、レイヤー構成、各クラスの責務、依存方向、共通処理の配置方針を整理する。

本書で定義した内容をもとに、後続のEntity設計、DTO設計、Repository設計、Service設計、Controller設計へ進む。

## 2. 対象範囲

本書の対象範囲は、タスク管理機能の初期MVPにおけるバックエンド構成とする。

対象に含めるものは以下とする。

- パッケージ構成
- レイヤー構成
- Controller層の責務
- Service層の責務
- Repository層の責務
- Entityの配置方針
- DTOの配置方針
- 認証・認可関連の配置方針
- 例外処理の配置方針
- 共通処理の配置方針
- バリデーションの配置方針

対象外とするものは以下とする。

- 各Entityの詳細項目定義
- 各DTOの詳細項目定義
- 各APIの詳細リクエスト・レスポンス定義
- SQLやマイグレーションファイルの詳細
- フロントエンド実装方針

## 4. バックエンド構成方針

バックエンドは、役割ごとに責務を分けたレイヤード構成とする。

基本的な構成は以下とする。

| レイヤー | 主な責務 |
|---|---|
| Controller | APIリクエストの受付、入力値の受け取り、レスポンス返却 |
| Service | 業務ルール、トランザクション、認可判定、複数Repositoryの制御 |
| Repository | DBアクセス |
| Entity | DBテーブルに対応するデータ構造 |
| DTO | APIリクエスト、APIレスポンス用のデータ構造 |
| Security | 認証、認可、セッション、CSRF制御 |
| Exception | 例外とエラーレスポンス制御 |
| Common | 共通定数、共通処理、共通レスポンスなど |

## 5. 依存方向

依存方向は以下を基本とする。

ControllerはServiceに依存する。

ServiceはRepository、Entity、DTOに依存する。

RepositoryはEntityに依存する。

EntityはControllerやServiceに依存しない。

DTOはAPI入出力用とし、Entityと混同しない。

依存方向は以下のように整理する。

```text
Controller
    ↓
Service
    ↓
Repository
    ↓
Entity
```

## 6. パッケージ構成方針

パッケージは、機能単位と責務単位を組み合わせて整理する。

初期MVPでは、以下のような構成を基本とする。

```text
backend
├── common
├── config
├── security
├── exception
├── auth
├── user
├── project
├── projectmember
├── invitation
├── task
├── comment
├── attachment
└── passwordreset
```

## 7. 共通パッケージ方針

`common` パッケージには、複数機能で共通利用するものを配置する。

配置候補は以下とする。

- 共通レスポンス形式
- ページング情報
- 共通定数
- 共通列挙値
- 日時処理の共通方針
- 作成日時、更新日時、削除日時などの監査項目に関する共通方針
- 論理削除対象Entityの共通項目方針

ただし、特定機能に強く依存する処理は `common` に置かず、対象機能のパッケージ内に置く。

## 8. configパッケージ方針

`config` パッケージには、アプリケーション全体の設定クラスを配置する。

配置候補は以下とする。

- Security設定
- CORS設定
- JSON設定
- ファイルアップロード設定
- メール送信設定
- JPA関連設定

設定クラスは、業務ロジックを持たない。

## 9. securityパッケージ方針

`security` パッケージには、認証・認可・セッション・CSRFに関する処理を配置する。

配置候補は以下とする。

- ログイン認証処理
- 認証ユーザー情報の取得処理
- ロール判定処理
- CSRFトークン取得API関連処理
- 認可チェック補助処理

認可判定はControllerだけで完結させず、Service層でも必要な業務ルールとして確認する。

## 10. exceptionパッケージ方針

`exception` パッケージには、例外とエラーレスポンス制御を配置する。

配置候補は以下とする。

- 業務例外
- 認可例外
- リソース未存在例外
- 入力チェック例外
- グローバル例外ハンドリング
- 共通エラーレスポンス変換

APIでは、エラー時に共通エラーレスポンス形式で返却する。

## 11. 機能別パッケージ方針

各機能パッケージには、その機能に関するController、Service、Repository、Entity、DTOを配置する。

例として、タスク管理に関するものは `task` パッケージにまとめる。

| パッケージ | 主な対象 |
|---|---|
| auth | ログイン、ログアウト、ログインユーザー取得 |
| user | ユーザー情報、システムロール |
| project | プロジェクト作成、更新、削除、一覧 |
| projectmember | プロジェクトメンバー、ロール変更 |
| invitation | 招待作成、招待承認 |
| task | タスク作成、更新、移動、削除 |
| comment | コメント作成、更新、削除 |
| attachment | 添付ファイル追加、取得、削除 |
| passwordreset | パスワードリセット申請、再設定 |

## 12. Controller設計方針

Controllerは、HTTPリクエストを受け取り、Serviceを呼び出し、HTTPレスポンスを返却する責務を持つ。

Controllerで行うことは以下とする。

- リクエストパス、HTTPメソッドの定義
- リクエストDTOの受け取り
- 入力チェックの起点
- ログインユーザー情報の受け取り
- Serviceの呼び出し
- レスポンスDTOの返却

Controllerで行わないことは以下とする。

- 業務ルールの判定
- DBアクセス
- 複雑な認可判定
- Entityの直接操作
- トランザクション制御

## 13. Service設計方針

Serviceは、業務ルールを実装する中心のレイヤーとする。

Serviceで行うことは以下とする。

- 業務ルールの判定
- 認可判定
- トランザクション制御
- Repositoryの呼び出し
- Entityの生成・更新
- DTOへの変換方針の制御
- 論理削除の制御
- メール送信処理の呼び出し

Serviceでは、システム管理者が0人になる操作の禁止、プロジェクト管理者が0人になる操作の禁止、削除権限、コメント編集権限、添付ファイル制約などの業務ルールを扱う。

## 14. Repository設計方針

Repositoryは、DBアクセスを担当する。

Repositoryで行うことは以下とする。

- Entityの取得
- Entityの保存
- Entityの検索
- 論理削除されていないデータの取得
- 一意制約確認に必要な検索

Repositoryで行わないことは以下とする。

- 業務ルールの判定
- 認可判定
- APIレスポンス生成
- メール送信

通常の取得では、論理削除済みデータを除外する。

## 15. Entity配置方針

EntityはDBテーブルに対応する。

Entityには、DB上で管理する項目、リレーション、作成日時、更新日時、削除日時などを定義する。

EntityはAPIレスポンスとして直接返却しない。

Entityの詳細項目は、後続のEntity設計書で定義する。

## 16. DTO配置方針

DTOはAPIのリクエストとレスポンスに使用する。

DTOは以下の種類に分ける。

- Request DTO
- Response DTO
- 一覧用DTO
- 詳細用DTO
- 内部処理用DTO

Request DTOは、入力チェックの対象とする。

Response DTOは、画面に返却する項目だけを持つ。

DTOの詳細項目は、後続のDTO設計書で定義する。

## 17. 認証・認可の配置方針

認証はSpring Securityを利用して行う。

ログイン状態はセッションで管理する。

認可は、システムロールとプロジェクトロールを組み合わせて判定する。

主なロールは以下とする。

| 種別 | ロール |
|---|---|
| システムロール | SYSTEM_ADMIN / USER |
| プロジェクトロール | PROJECT_ADMIN / MEMBER / VIEWER |

プロジェクト単位の権限判定は、Service層で対象プロジェクトに対するロールを確認する。

## 18. トランザクション方針

データ更新を行うServiceメソッドでは、トランザクションを使用する。

対象となる操作は以下とする。

- プロジェクト作成
- プロジェクト更新
- プロジェクト削除
- メンバー招待
- ロール変更
- タスク作成
- タスク更新
- タスクステータス変更
- タスク削除
- コメント作成
- コメント更新
- コメント削除
- 添付ファイル追加
- 添付ファイル削除
- パスワードリセット
- 招待承認

参照のみの処理では、読み取り専用トランザクションを検討する。

## 19. 論理削除方針

主要データの削除は論理削除とする。

論理削除では、対象データの `deleted_at` に削除日時を設定する。

通常の取得処理では、 `deleted_at` が設定されているデータを取得対象外とする。

ユーザー削除時は、作成済みタスクやコメントは削除せず、表示名を `削除済ユーザー` として扱う。

プロジェクト削除時は、プロジェクト、プロジェクトメンバー、タスク、コメント、添付ファイルを論理削除する。

招待情報とパスワードリセット情報は履歴として残す。

## 20. バリデーション方針

入力チェックは、Request DTOを起点として行う。

形式チェック、必須チェック、文字数チェックなどはDTOで扱う。

業務ルールに関わるチェックはServiceで扱う。

例は以下とする。

| チェック内容 | 主な配置 |
|---|---|
| 必須入力 | Request DTO |
| 文字数 | Request DTO |
| メール形式 | Request DTO |
| 添付ファイル容量 | Service |
| 添付ファイル種別 | Service |
| 権限チェック | Service |
| 管理者0人防止 | Service |
| 招待URL有効期限 | Service |
| パスワードリセットURL有効期限 | Service |

## 21. メール送信処理方針

メール送信は、招待メールとパスワードリセットメールで使用する。

メール送信処理はServiceから呼び出す。

開発環境ではMailpitを使用する。

本番向けメール送信方式は後続工程で具体化する。

## 22. 添付ファイル処理方針

添付ファイル本体はDBに保存する。

添付ファイル処理では以下を確認する。

- 1ファイルあたり5MB以内であること
- 許可されたファイル種別であること
- 対象タスクが存在すること
- 操作ユーザーに権限があること

添付ファイル取得APIでは、ファイルデータを返却する。

添付ファイル追加・削除は、Service層で権限と制約を確認する。

## 23. エラーレスポンス方針

エラー時は、共通エラーレスポンス形式で返却する。

主なHTTPステータスは以下とする。

| ステータス | 用途 |
|---|---|
| 400 | 入力不正 |
| 401 | 未認証 |
| 403 | 権限不足、CSRF不正 |
| 404 | 対象データなし |
| 409 | 業務ルール違反、競合 |
| 500 | サーバーエラー |

例外の詳細設計は、後続の例外設計書で定義する。

## 24. 命名方針

クラス名は、役割が分かる名前にする。

| 種別 | 命名方針 |
|---|---|
| Controller | 対象名 + Controller |
| Service | 対象名 + Service |
| Repository | 対象名 + Repository |
| Entity | テーブルに対応する単数名 |
| Request DTO | 操作名 + Request |
| Response DTO | 操作名 + Response |
| Exception | 内容 + Exception |

詳細な命名は、各詳細設計書で具体化する。

## 26. 決定事項

- バックエンドはレイヤード構成とする
- Controller、Service、Repository、Entity、DTOの責務を分ける
- 業務ルールはService層に集約する
- DBアクセスはRepository層に集約する
- API入出力にはDTOを使用する
- EntityはAPIレスポンスとして直接返却しない
- 認可判定はService層で必ず行う
- 主要データの削除は論理削除とする
- 開発用メール確認にはMailpitを使用する
