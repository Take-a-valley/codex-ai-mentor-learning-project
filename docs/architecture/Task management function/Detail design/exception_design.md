# 例外設計書

## 1. 本書の目的

本書は、タスク管理機能の初期MVPにおける例外設計を定義することを目的とする。

例外設計では、バックエンドで発生するエラーを分類し、HTTPステータス、エラーコード、エラーメッセージ、エラーレスポンス形式を統一する。

Controller、Service、Repositoryで発生した例外を共通的に扱うことで、フロントエンドが一貫した形式でエラーを処理できるようにする。

## 2. 対象範囲

対象に含めるものは以下とする。

- 例外分類
- エラーレスポンス形式
- HTTPステータスの使い分け
- エラーコード方針
- 入力チェックエラー
- 認証エラー
- 認可エラー
- 対象データなしエラー
- 業務ルール違反
- ファイル制約違反
- CSRFエラー
- 想定外エラー
- 共通例外ハンドリング方針

対象外とするものは以下とする。

- 具体的なJavaコード
- Controllerの実装詳細
- Serviceの実装詳細
- Spring Security設定詳細
- ログ出力の実装詳細
- フロントエンドでのエラー表示実装

## 3. 例外設計方針

APIのエラーレスポンスは共通形式で返却する。

Controllerごとに独自のエラーレスポンス形式は作らない。

Service層で検出した業務ルール違反は、共通例外へ変換して扱う。

入力チェックエラーは、対象項目が分かるようにdetailsへ格納する。

認証・認可・CSRFに関するエラーは、Spring Securityと共通例外ハンドリングで扱う。

想定外エラーでは、内部エラーの詳細をレスポンスに含めない。

エラーメッセージは、フロントエンドでそのまま表示しても違和感がない内容を基本とする。

## 4. エラーレスポンス形式

エラー発生時は、DTO設計書で定義したErrorResponseを返却する。

### ErrorResponse

| 項目 | 型 | 概要 |
|---|---|---|
| code | string | エラーコード |
| message | string | エラーメッセージ |
| details | array | 入力項目ごとの詳細エラー |
| timestamp | string | エラー発生日時 |

### ErrorDetail

| 項目 | 型 | 概要 |
|---|---|---|
| field | string | エラー対象項目 |
| message | string | 項目別エラーメッセージ |

入力チェックエラー以外では、detailsは空配列を基本とする。

## 5. HTTPステータス方針

| HTTPステータス | 用途 |
|---|---|
| 400 Bad Request | 入力値不正、リクエスト形式不正 |
| 401 Unauthorized | 未認証 |
| 403 Forbidden | 権限不足、CSRF不正 |
| 404 Not Found | 対象データなし |
| 409 Conflict | 業務ルール違反、競合状態 |
| 413 Payload Too Large | 添付ファイル容量超過 |
| 500 Internal Server Error | 想定外エラー |

## 6. 例外分類

| 分類 | 概要 | 主なHTTPステータス |
|---|---|---|
| 入力チェックエラー | 必須、形式、文字数などの入力不備 | 400 |
| 認証エラー | ログインしていない、セッションが無効 | 401 |
| 認可エラー | 操作権限がない | 403 |
| CSRFエラー | CSRFトークンがない、または不正 | 403 |
| 対象データなしエラー | 指定IDのデータが存在しない | 404 |
| 業務ルール違反 | 管理者0人禁止、期限切れtokenなど | 409 |
| ファイル制約違反 | 容量超過、許可されないファイル種別 | 400 / 413 |
| 想定外エラー | 未分類のサーバーエラー | 500 |

## 7. 共通エラーコード

| エラーコード | HTTPステータス | 概要 |
|---|---|---|
| VALIDATION_ERROR | 400 | 入力チェックエラー |
| INVALID_REQUEST | 400 | リクエスト形式不正 |
| UNAUTHENTICATED | 401 | 未認証 |
| ACCESS_DENIED | 403 | 権限不足 |
| CSRF_INVALID | 403 | CSRFトークン不正 |
| RESOURCE_NOT_FOUND | 404 | 対象データなし |
| BUSINESS_RULE_VIOLATION | 409 | 業務ルール違反 |
| CONFLICT | 409 | データ競合 |
| FILE_TOO_LARGE | 413 | 添付ファイル容量超過 |
| INVALID_FILE_TYPE | 400 | 許可されないファイル種別 |
| TOKEN_INVALID | 400 | token不正 |
| TOKEN_EXPIRED | 409 | token期限切れ |
| INTERNAL_SERVER_ERROR | 500 | 想定外エラー |

## 8. 入力チェックエラー

入力チェックエラーは、Request DTOの入力値が不正な場合に発生する。

主な発生条件は以下とする。

| 発生条件 | HTTPステータス | エラーコード |
|---|---|---|
| 必須項目が未入力 | 400 | VALIDATION_ERROR |
| メールアドレス形式が不正 | 400 | VALIDATION_ERROR |
| 文字数上限を超過 | 400 | VALIDATION_ERROR |
| パスワード形式が不正 | 400 | VALIDATION_ERROR |
| enum値に存在しない値 | 400 | VALIDATION_ERROR |

入力チェックエラーでは、detailsに項目単位のエラーを設定する。

## 9. 認証エラー

認証エラーは、認証が必要なAPIに未ログイン状態でアクセスした場合に発生する。

| 発生条件 | HTTPステータス | エラーコード |
|---|---|---|
| 未ログイン | 401 | UNAUTHENTICATED |
| セッション期限切れ | 401 | UNAUTHENTICATED |
| 削除済みユーザーによるアクセス | 401 | UNAUTHENTICATED |

セッション有効期限は、最終アクセスから120分とする。

## 10. 認可エラー

認可エラーは、ログイン済みだが操作権限がない場合に発生する。

| 発生条件 | HTTPステータス | エラーコード |
|---|---|---|
| SYSTEM_ADMINのみ可能な操作を通常ユーザーが実行 | 403 | ACCESS_DENIED |
| PROJECT_ADMINのみ可能な操作をMEMBERが実行 | 403 | ACCESS_DENIED |
| MEMBER以上が必要な操作をVIEWERが実行 | 403 | ACCESS_DENIED |
| 対象プロジェクトに参加していないユーザーがアクセス | 403 | ACCESS_DENIED |
| コメント投稿者本人以外がコメント更新 | 403 | ACCESS_DENIED |

認可判定はService層で行う。

## 11. CSRFエラー

CSRFエラーは、状態変更APIでCSRFトークンが不足または不正な場合に発生する。

| 発生条件 | HTTPステータス | エラーコード |
|---|---|---|
| CSRFトークン未送信 | 403 | CSRF_INVALID |
| CSRFトークン不正 | 403 | CSRF_INVALID |
| CSRFトークン期限切れ | 403 | CSRF_INVALID |

CSRFトークンの生成と検証はSpring Securityで行う。

## 12. 対象データなしエラー

対象データなしエラーは、指定されたIDやtokenに対応するデータが存在しない場合に発生する。

| 発生条件 | HTTPステータス | エラーコード |
|---|---|---|
| projectIdに対応するProjectが存在しない | 404 | RESOURCE_NOT_FOUND |
| taskIdに対応するTaskが存在しない | 404 | RESOURCE_NOT_FOUND |
| commentIdに対応するTaskCommentが存在しない | 404 | RESOURCE_NOT_FOUND |
| attachmentIdに対応するAttachmentが存在しない | 404 | RESOURCE_NOT_FOUND |
| memberIdに対応するProjectMemberが存在しない | 404 | RESOURCE_NOT_FOUND |
| tokenに対応する招待情報が存在しない | 404 | RESOURCE_NOT_FOUND |
| tokenに対応するパスワードリセット情報が存在しない | 404 | RESOURCE_NOT_FOUND |

論理削除済みデータは、通常操作では存在しないものとして扱う。

## 13. 業務ルール違反

業務ルール違反は、入力形式は正しいが、業務上許可できない操作を行った場合に発生する。

| 発生条件 | HTTPステータス | エラーコード |
|---|---|---|
| PROJECT_ADMINが0人になるロール変更 | 409 | BUSINESS_RULE_VIOLATION |
| PROJECT_ADMINが0人になるメンバー削除 | 409 | BUSINESS_RULE_VIOLATION |
| 同一プロジェクト・同一メールのPENDING招待が存在する | 409 | CONFLICT |
| 承認済み招待を再度承認 | 409 | BUSINESS_RULE_VIOLATION |
| 期限切れ招待を承認 | 409 | TOKEN_EXPIRED |
| 無効化済み招待を承認 | 409 | BUSINESS_RULE_VIOLATION |
| 招待先メールアドレスとログインユーザーのメールアドレスが不一致 | 409 | BUSINESS_RULE_VIOLATION |
| 使用済みパスワードリセットtokenを使用 | 409 | BUSINESS_RULE_VIOLATION |
| 期限切れパスワードリセットtokenを使用 | 409 | TOKEN_EXPIRED |
| 論理削除済みプロジェクト配下のデータを操作 | 409 | BUSINESS_RULE_VIOLATION |
| 論理削除済みタスクにコメントや添付ファイルを追加 | 409 | BUSINESS_RULE_VIOLATION |
| 未登録ユーザー登録時に招待先emailのユーザーが既に存在する | 409 | CONFLICT |

## 14. ファイル制約違反

添付ファイル追加時には、容量とファイル種別を検証する。

| 発生条件 | HTTPステータス | エラーコード |
|---|---|---|
| ファイル未指定 | 400 | VALIDATION_ERROR |
| 1ファイルあたり5MBを超過 | 413 | FILE_TOO_LARGE |
| 許可されないファイル種別 | 400 | INVALID_FILE_TYPE |
| ファイル名が不正 | 400 | VALIDATION_ERROR |

添付可能なファイル種別は、画像、PDF、テキスト、Office系ファイルとする。

## 15. 想定外エラー

想定外エラーは、事前に分類できないサーバー側のエラーである。

| 発生条件 | HTTPステータス | エラーコード |
|---|---|---|
| 未分類のRuntimeException | 500 | INTERNAL_SERVER_ERROR |
| DB接続障害 | 500 | INTERNAL_SERVER_ERROR |
| メール送信処理の想定外エラー | 500 | INTERNAL_SERVER_ERROR |
| ファイル取得処理の想定外エラー | 500 | INTERNAL_SERVER_ERROR |

想定外エラーでは、内部例外の詳細をAPIレスポンスに含めない。

詳細情報はサーバーログに出力する。

## 17. 共通例外ハンドリング方針

例外は共通例外ハンドリングでErrorResponseへ変換する。

Controllerでは、個別にエラーレスポンスを組み立てない。

Serviceでは、業務ルール違反や対象データなしを検出し、例外として通知する。

Repositoryでは、DBアクセスに専念し、業務ルール例外は原則として発生させない。

## 18. レイヤー別の責務

| レイヤー | 例外に関する責務 |
|---|---|
| Controller | Request DTOの受け取り、入力チェックの起点 |
| Service | 業務ルール違反、認可、対象データなしの検出 |
| Repository | DBアクセス、取得結果の返却 |
| 共通例外ハンドラ | 例外をErrorResponseへ変換 |
| Spring Security | 認証、CSRF、セッション関連の例外検出 |

## 19. エラーメッセージ方針

エラーメッセージは、利用者に伝えて問題ない内容にする。

内部実装、SQL、スタックトレース、クラス名はレスポンスに含めない。

セキュリティ上、存在有無を推測されると困る情報はメッセージに含めない。

パスワードリセット申請では、メールアドレスが登録済みかどうかを返却しない。

## 20. ログ出力方針

業務ルール違反や入力チェックエラーは、必要最低限のログに留める。

想定外エラーは、調査に必要な情報をサーバーログへ出力する。

パスワード、token、CSRFトークン、セッションIDはログに出力しない。

個人情報をログに出力する場合は、必要最小限にする。

## 21. 決定事項

- エラーレスポンスはErrorResponse形式で統一する
- 入力チェックエラーはdetailsに項目単位で設定する
- Controllerごとに独自のエラー形式は作らない
- 認可判定はService層で行う
- 論理削除済みデータは通常操作では存在しないものとして扱う
- 想定外エラーでは内部詳細をレスポンスに含めない
- パスワードリセット申請ではメールアドレスの存在有無を返却しない
- tokenやセッションIDなどの機密情報はログに出力しない
