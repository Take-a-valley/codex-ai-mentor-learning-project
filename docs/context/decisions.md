# Decisions

| 日付 | 決定事項 | 判断理由 | 影響範囲 |
|---|---|---|---|
| 未記録 | AIは実装代行ではなくメンターとして使う | 学習効果を重視するため | 開発プロセス全体 |
| 未記録 | 設計書は `/docs/architecture/` にユーザーが作成する | 設計力を身につけるため | ドキュメント管理 |
| 2026-07-01 | 実装開始前の最終整理では、実装開始に必要な未決事項のみ決める | 実装前にすべてを決め切るより、初期MVP開始に必要な判断へ絞るため | 詳細設計完了後の実装準備 |
| 2026-07-01 | Next.jsはApp Routerを採用する方針 | Next.js 16系ではApp Routerが主流で、今後の標準構成として学びやすいため | フロントエンド構成、ルーティング |
| 2026-07-01 | shadcn/uiは必要箇所のみ使用する方針 | Tailwind CSSを基本にしつつ、フォーム、モーダル、ダイアログ、テーブルなどを効率よく実装するため | フロントエンドUI |
| 2026-07-01 | パスワードルールは8文字以上72文字以下、英字と数字を各1文字以上、記号任意を推奨方針とする | 初期MVPとして実装しやすく、最低限の強度を確保するため | 認証、入力チェック、テスト |
| 2026-07-01 | ログイン失敗回数制限は初期MVP対象外とし、将来拡張とする方針 | アカウントロックや解除設計まで範囲が広がるため | 認証、セキュリティ |
| 2026-07-01 | 招待・パスワードリセットの期限切れ処理は参照時に判定する方針 | バッチ処理を作らず初期MVPとして実装しやすいため | 招待、パスワードリセット |
| 2026-07-01 | JavaのID型はLong、日時はLocalDateTime、期限日はLocalDateを基本方針とする | Spring Boot / JPAで扱いやすく、業務データとして自然なため | Entity、DTO、Repository |
| 2026-07-01 | tokenは推測困難なランダム文字列としてDB保存する方針 | 初期MVPでは学習・実装難度を抑えるため | 招待、パスワードリセット |
| 2026-07-01 | ControllerではSpring Securityの認証情報からログインユーザーを取得し、ServiceへログインユーザーIDまたは情報を渡す方針 | Service層で認可判定しやすくするため | Controller、Service、Security |
| 2026-07-01 | 一般的な開発順序に準じ、環境・DB・バックエンド基盤から実装を開始する方針 | 業務アプリ開発の一般的な流れに沿い、土台を安定させるため | 実装計画 |
